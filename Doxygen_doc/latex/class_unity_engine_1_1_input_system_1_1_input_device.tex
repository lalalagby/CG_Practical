\doxysubsection{Detailed Description}
Represents an input device which is always the root of a hierarchy of \doxylink{class_unity_engine_1_1_input_system_1_1_input_control}{Input\+Control} instances. 

Input devices act as the container for control hierarchies. Every hierarchy has to have a device at the root. Devices cannot occur as children of other controls.

Devices are usually created automatically in response to hardware being discovered by the \doxylink{namespace_unity}{Unity} runtime. However, it is possible to manually add devices using methods such as Input\+System.\+Add\+Device$<$\+TDevice$>$(string).

\end{DoxyCode}
 

There are subclasses representing the most common types of devices, like \doxylink{class_unity_engine_1_1_input_system_1_1_mouse}{Mouse}, \doxylink{class_unity_engine_1_1_input_system_1_1_keyboard}{Keyboard}, \doxylink{class_unity_engine_1_1_input_system_1_1_gamepad}{Gamepad}, and \doxylink{class_unity_engine_1_1_input_system_1_1_touchscreen}{Touchscreen}.

To create your own types of devices, you can derive from \doxylink{class_unity_engine_1_1_input_system_1_1_input_device}{Input\+Device} and register your device as a new "{}layout"{}.

For a more complete example of how to implement custom input devices, check out the "{}\+Custom Device"{} sample which you can install from the \doxylink{namespace_unity}{Unity} package manager.

And, as always, you can also find more information in the \href{../manual/Devices.html}{\texttt{ manual}}.

\doxysubsubsection{\texorpdfstring{MakeCurrent()}{MakeCurrent()}}
{\footnotesize\ttfamily \label{class_unity_engine_1_1_input_system_1_1_input_device_a0a98203c32eb180ab8b7c862dc27635c} 
virtual void Unity\+Engine.\+Input\+System.\+Input\+Device.\+Make\+Current (\begin{DoxyParamCaption}{}{}\end{DoxyParamCaption})\hspace{0.3cm}{\ttfamily [inline]}, {\ttfamily [virtual]}}



Make this the current device of its type. 

This method is called automatically by the input system when a device is added or when input is received on it. Many types of devices have {\ttfamily .current} getters that allow querying the last used device of a specific type directly (for example, see \doxylink{class_unity_engine_1_1_input_system_1_1_gamepad_a6f0bd4b71340da07b56f6ec56097da1d}{Gamepad.\+current}).

There is one special case, however, related to noise. A device that has noisy controls (i.\+e. controls for which \doxylink{class_unity_engine_1_1_input_system_1_1_input_control_afa5a8531d7c9a030e8e484560a59fe6f}{Input\+Control.\+noisy} is true) may receive input events that contain no meaningful user interaction but are simply just noise from the device. A good example of this is the PS4 gamepad which has a built-\/in gyro and may thus constantly feed events into the input system even if not being actually in use. If, for example, an Xbox gamepad and PS4 gamepad are both connected to a PC and the user is playing with the Xbox gamepad, the PS4 gamepad would still constantly make itself \doxylink{class_unity_engine_1_1_input_system_1_1_gamepad_a6f0bd4b71340da07b56f6ec56097da1d}{Gamepad.\+current} by simply flooding the system with events. Hence why by default, noise on {\ttfamily .current} getters will be filtered out and a device will only see {\ttfamily Make\+Current} getting called if there input was detected on non-\/noisy controls.

If true, the device is capable of delivering input while the application is running in the background, i.\+e. while {\ttfamily Application.\+is\+Focused} is false. 

TODO\+: rename this to can\+Receive\+Input\+In\+Background (once we can break API)

Whether the device can generate input while in the background.

The value of this property is determined by three separator factors.

For one, \doxylink{class_unity_engine_1_1_input_system_1_1_input_device_a1b56ee045064190e84a9c80098da6f33}{native} devices have an inherent value for this property that can be retrieved through Query\+Can\+Run\+In\+Background. This determines whether at the input collection level, the device is capable of producing input independent of application. This is rare and only a select set of hardware, platform, and SDK/\+API combinations support this. The prominent class of input devices that in general do support this behavior are VR devices.

Furthermore, the property may be force-\/set through a device\textquotesingle{}s \doxylink{class_unity_engine_1_1_input_system_1_1_input_control_a8d349861ff83c7a74573bea9a2cf2886}{Input\+Control.\+layout} by means of \doxylink{class_unity_engine_1_1_input_system_1_1_layouts_1_1_input_control_layout_a23c44ec65df0296019186860b8cf6c5c}{Input\+Control\+Layout.\+can\+Run\+In\+Background}.

Lastly, in the editor, the value of the property may be overridden depending on \doxylink{class_unity_engine_1_1_input_system_1_1_input_settings_a92a36004e55e36001de4663d29d68094}{Input\+Settings.\+editor\+Input\+Behavior\+In\+Play\+Mode} in case certain devices are automatically kept running in play mode even when no Game View has focus.

Be aware that as far as players are concerned, only certain platforms support running \doxylink{namespace_unity}{Unity} while not having focus. On mobile platforms, for example, this is generally not supported. In this case, the value of this property has no impact on input while the application does not have focus. See \doxylink{class_unity_engine_1_1_input_system_1_1_input_settings_a11467cec652b53340f93f7a788dac2d5}{Input\+Settings.\+background\+Behavior} for more details.

A device that is disabled will not receive events. I.\+e. events that are being sent to the device will be ignored.

When disabling a \doxylink{class_unity_engine_1_1_input_system_1_1_input_device_a1b56ee045064190e84a9c80098da6f33}{native} device, a \doxylink{}{disable command} will also be sent to the \doxylink{}{runtime}. This is especially important for devices such as \doxylink{class_unity_engine_1_1_input_system_1_1_sensor}{sensors} that incur both computation and battery consumption overhead while enabled.

Specific types of devices can choose to start out in disabled state by default. This is generally the case for \doxylink{class_unity_engine_1_1_input_system_1_1_sensor}{sensors} to ensure that their overhead is only incurred when actually being used by the application.

Whether the device comes from the \doxylink{}{runtime} 

Whether the device has been discovered by the \doxylink{namespace_unity}{Unity} runtime.

Devices can be discovered when \doxylink{}{reported} by the runtime or they can be added manually through the various \doxylink{ab8ab06ad1c724c2d188608d805eb4009}{Add\+Device} APIs. Devices reported by the runtime will return true for this property whereas devices added manually will return false.

Devices reported by the runtime will usually come from the \doxylink{namespace_unity}{Unity} engine itself.

Whether the device requires an extra update before rendering. 

The value of this property is determined by Input\+Control\+Layout.\+update\+Before\+Render in the device\textquotesingle{}s \doxylink{}{control layout}.

The extra update is necessary for tracking devices that are used in rendering code. For example, the eye transforms of an HMD should be refreshed right before rendering as refreshing only in the beginning of the frame will lead to a noticeable lag.

 When you build your project, \doxylink{namespace_burst}{Burst} compiles your code, then creates a single dynamic library, and puts it into the Plugins folder for the platform you\textquotesingle{}re targeting. For example, on Windows, the path is {\ttfamily Data/\+Plugins/lib\+\_\+burst\+\_\+generated.\+dll}.

\texorpdfstring{$>$}{>}\mbox{[}!\+NOTE\mbox{]} \texorpdfstring{$>$}{>}This is different if your target platform is i\+OS. Instead, \doxylink{namespace_unity}{Unity} generates a static library because of Apple\textquotesingle{}s submission requirements for Test\+Flight.

The job system runtime loads the generated library the first time a \doxylink{namespace_burst}{Burst} compiled method is invoked.

To control \doxylink{namespace_burst}{Burst}\textquotesingle{}s AOT compilation, use the settings in the {\bfseries{\doxylink{namespace_burst}{Burst} AOT Settings}} section of the \doxylink{class_player}{Player} Settings window ({\bfseries{Edit \texorpdfstring{$>$}{>} \doxylink{class_player}{Player} Settings \texorpdfstring{$>$}{>} \doxylink{namespace_burst}{Burst} AOT Settings}}). For more information, see \doxysectlink{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8burst_0d1_88_87_2_documentation_0i_2building-aot-settings}{Burst AOT Settings reference}{0}.

If you\textquotesingle{}re compiling for a non-\/desktop platform, then \doxylink{namespace_burst}{Burst} compilation requires specific platform compilation tools (similar to \href{https://docs.unity3d.com/Manual/IL2CPP.html}{\texttt{ IL2\+CPP}}). By default, desktop platforms (mac\+OS, Linux, Windows) don\textquotesingle{}t need external toolchain support, unless you enable the {\bfseries{Use Platform SDK Linker}} setting in the \doxysectlink{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8burst_0d1_88_87_2_documentation_0i_2building-aot-settings}{Burst AOT Settings}{0}.

If you select an invalid target (one with missing tools, or unsupported), \doxylink{namespace_unity}{Unity} doesn\textquotesingle{}t use \doxylink{namespace_burst}{Burst} compilation, which might lead it to fail, but \doxylink{namespace_unity}{Unity} still builds the target without \doxylink{namespace_burst}{Burst} optimizations.

\texorpdfstring{$>$}{>}\mbox{[}!\+NOTE\mbox{]} \texorpdfstring{$>$}{>}\doxylink{namespace_burst}{Burst} supports cross-\/compilation between desktop platforms (mac\+OS/\+Linux/\+Windows) by default.

The maximum target CPU is hardcoded per platform. For standalone builds that target desktop platforms (Windows/\+Linux/mac\+OS) you can choose the supported targets via the \doxysectlink{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8burst_0d1_88_87_2_documentation_0i_2building-aot-settings}{Burst AOT Settings}{0}\hypertarget{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8burst_0d1_88_87_2_documentation_0i_2building-projects_autotoc_md292}{}\doxysubsection{\texorpdfstring{Projects that don\textquotesingle{}t use Burst}{Projects that don\textquotesingle{}t use Burst}}\label{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8burst_0d1_88_87_2_documentation_0i_2building-projects_autotoc_md292}
Some projects can\textquotesingle{}t use \doxylink{namespace_burst}{Burst} as the compiler\+:


\begin{DoxyItemize}
\item i\+OS projects from the Windows Editor
\item Android projects from the Linux Editor
\item Xcode projects generated from the {\bfseries{Create Xcode Project}} option
\end{DoxyItemize}\hypertarget{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8burst_0d1_88_87_2_documentation_0i_2building-projects_autotoc_md293}{}\doxysection{\texorpdfstring{Multiple Burst targets}{Multiple Burst targets}}\label{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8burst_0d1_88_87_2_documentation_0i_2building-projects_autotoc_md293}
When \doxylink{namespace_burst}{Burst} compiles multiple target platforms during a build, it has to perform separate compilations. For example, if you want to compile {\ttfamily X64\+\_\+\+SSE2} and {\ttfamily X64\+\_\+\+SSE4}, the \doxylink{namespace_burst}{Burst} has to do two separate compilations to generate code for each of the targets you choose.

To keep the combinations of targets to a minimum, \doxylink{namespace_burst}{Burst} target platforms require multiple processor instruction sets underneath\+:


\begin{DoxyItemize}
\item {\ttfamily SSE4.\+2} is gated on having {\ttfamily SSE4.\+2} and {\ttfamily POPCNT} instruction sets.
\item {\ttfamily AVX2} is gated on having {\ttfamily AVX2}, {\ttfamily FMA}, {\ttfamily F16C}, {\ttfamily BMI1}, and {\ttfamily BMI2} instruction sets.
\item {\ttfamily ARMV8A} is a basic Armv8-\/A CPU target
\item {\ttfamily ARMV8\+A\+\_\+\+HALFFP} is {\ttfamily ARMV8A} plus the following extensions\+: {\ttfamily fullfp16}, {\ttfamily dotprod}, {\ttfamily crypto}, {\ttfamily crc}, {\ttfamily rdm}, {\ttfamily lse}. In practice, this means Cortex A75/\+A55 and later cores.
\item {\ttfamily ARMV9A} is {\ttfamily ARMV8\+A\+\_\+\+HALFFP} plus SVE2 support. In practice, this means Cortex X2/\+A710/\+A510 and later cores. {\bfseries{Important\+:}} this target is currently experimental.
\end{DoxyItemize}\hypertarget{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8burst_0d1_88_87_2_documentation_0i_2building-projects_autotoc_md294}{}\doxysection{\texorpdfstring{Dynamic dispatch based on runtime CPU features}{Dynamic dispatch based on runtime CPU features}}\label{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8burst_0d1_88_87_2_documentation_0i_2building-projects_autotoc_md294}
For all {\ttfamily x86}/{\ttfamily x64} CPU desktop platforms, as well as for 64-\/bit Arm on Android, \doxylink{namespace_burst}{Burst} takes into account the CPU features available at runtime to dispatch jobs to different versions it compiles.

For {\ttfamily x86} and {\ttfamily x64} CPUs, \doxylink{namespace_burst}{Burst} supports {\ttfamily SSE2} and {\ttfamily SSE4} instruction sets at runtime only.

For example, with dynamic CPU dispatch, if your CPU supports {\ttfamily SSE3} and below, \doxylink{namespace_burst}{Burst} selects {\ttfamily SSE2} automatically.

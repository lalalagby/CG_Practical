\doxysubsection{Detailed Description}
A chunk of memory signaling a data transfer in the input system. 

Input events are raw memory buffers akin to a byte array. For most uses of the input system, it is not necessary to be aware of the event stream in the background. Events are written to the internal event buffer by producers -- usually by the platform-\/specific backends sitting in the \doxylink{namespace_unity}{Unity} runtime. Once per fixed or dynamic update (depending on what \doxylink{class_unity_engine_1_1_input_system_1_1_input_settings_ab7d15182096158aaf407f0b7e9aed8ae}{Input\+Settings.\+update\+Mode} is set to), the input system then goes and flushes out the internal event buffer to process pending events.

Events may signal general device-\/related occurrences (such as \doxylink{struct_unity_engine_1_1_input_system_1_1_low_level_1_1_device_configuration_event}{Device\+Configuration\+Event} or \doxylink{struct_unity_engine_1_1_input_system_1_1_low_level_1_1_device_remove_event}{Device\+Remove\+Event}) or they may signal input activity. The latter kind of event is called "{}state events"{}. In particular, these events are either \doxylink{struct_unity_engine_1_1_input_system_1_1_low_level_1_1_state_event}{State\+Event}, only.

Events are solely focused on input. To effect output on an input device (e.\+g. haptics effects), "{}commands"{} (see \doxylink{struct_unity_engine_1_1_input_system_1_1_low_level_1_1_input_device_command}{Input\+Device\+Command}) are used.

Event processing can be listened to using Input\+System.\+on\+Event. This callback will get triggered for each event as it is processed by the input system.

Note that there is no "{}routing"{} mechanism for events, i.\+e. no mechanism by which the input system looks for a handler for a specific event. Instead, events represent low-\/level activity that the input system directly integrates into the state of its \doxylink{class_unity_engine_1_1_input_system_1_1_input_device}{Input\+Device} instances.

Each type of event is distinguished by its own Four\+CC type tag. The tag can be queried from the \doxylink{struct_unity_engine_1_1_input_system_1_1_low_level_1_1_input_event_a40a35f018657b0603e16a6210b8d5a19}{type} property.

Each event will receive a unique ID when queued to the internal event buffer. The ID can be queried using the \doxylink{struct_unity_engine_1_1_input_system_1_1_low_level_1_1_input_event_ae4c14fae911caca0e67f4088a972076b}{event\+Id} property. Over the lifetime of the input system, no two events will receive the same ID. If you repeatedly queue an event from the same memory buffer, each individual call of Input\+System.\+Queue\+Event will result in its own unique event ID.

All events are device-\/specific meaning that \doxylink{struct_unity_engine_1_1_input_system_1_1_low_level_1_1_input_event_a2b75e12cb1f2c24b77872636a72ecb4b}{device\+Id} will always reference some device (which, however, may or may not translate to an \doxylink{class_unity_engine_1_1_input_system_1_1_input_device}{Input\+Device}; that part depends on whether the input system was able to create an \doxylink{class_unity_engine_1_1_input_system_1_1_input_device}{Input\+Device} based on the information received from the backend).

 A C\# math library providing vector types and math functions with a shader like syntax. Used by the \doxylink{namespace_burst}{Burst} compiler to compile C\#/\+IL to highly efficient native code.

The main goal of this library is to provide a friendly Math API familiar to SIMD and graphic/shaders developers, using the well known {\ttfamily float4}, {\ttfamily float3} types...etc. with all intrinsics functions provided by a static class {\ttfamily math} that can be imported easily into your C\# program with {\ttfamily using static Unity.\+Mathematics.\+math}.

In addition to this, the \doxylink{namespace_burst}{Burst} compiler is able to recognize these types and provide the optimized SIMD type for the running CPU on all supported platforms (x64, ARMv7a...etc.)

NOTICE\+: The API is a work in progress and we may introduce breaking changes (API and underlying behavior)\hypertarget{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8mathematics_0d1_82_86_2_documentation_0i_2mathematics_autotoc_md1665}{}\doxysection{\texorpdfstring{Usage}{Usage}}\label{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8mathematics_0d1_82_86_2_documentation_0i_2mathematics_autotoc_md1665}

After years of feedback and experience with the previous API, we believe that providing an API that is closer to the way graphics developers have been using math libraries should better help its adoption and the ease of its usage. HLSL / GLSL math library is a very well designed, well understood math library leading to greater consistency.\hypertarget{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8mathematics_0d1_82_86_2_documentation_0i_2mathematics_autotoc_md1670}{}\doxysubsection{\texorpdfstring{Why not using {\ttfamily System.\+Numerics.\+Vectors}?}{Why not using {\ttfamily System.\+Numerics.\+Vectors}?}}\label{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8mathematics_0d1_82_86_2_documentation_0i_2mathematics_autotoc_md1670}
Mainly for the reason mentioned above, {\ttfamily System.\+Numerics.\+Vectors} is in many ways similar to our previous Vector library (more object oriented than graphics programming oriented). Also the fact that our \doxylink{namespace_burst}{Burst} compiler is able to recognize a lot more patterns for SIMD types and math intrinsics makes it easier to work with a dedicated API that reflects this ability.\hypertarget{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8mathematics_0d1_82_86_2_documentation_0i_2mathematics_autotoc_md1671}{}\doxysubsection{\texorpdfstring{Naming convention}{Naming convention}}\label{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8mathematics_0d1_82_86_2_documentation_0i_2mathematics_autotoc_md1671}
In C\# {\ttfamily int} and {\ttfamily float} are considered builtin types. \doxylink{namespace_burst}{Burst} extends this set of bultin types to also include vectors, matrices and quaternions. These types are bultin in the sense that \doxylink{namespace_burst}{Burst} knows about them and is be able to generate better code using these types than what would be possible with equivalent code using custom types.

To signify that these types are bultin their type names are in all lower case. The operators on these bultin types found in {\ttfamily Unity.\+Mathematics.\+math} are considered intrinsics and are thus always in lower case.

There are no plans to extend the set of intrinsic types beyond the current set of vectors ({\ttfamily typeN}), matrices ({\ttfamily type\+NxN}) and quaternions ({\ttfamily quaternion}).

This convention has the added benefit of making the library highly compatible with shader code and makes porting or sharing code between the two almost frictionless.\hypertarget{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8mathematics_0d1_82_86_2_documentation_0i_2mathematics_autotoc_md1672}{}\doxysubsection{\texorpdfstring{Why can\textquotesingle{}t we send a PR yet?}{Why can\textquotesingle{}t we send a PR yet?}}\label{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8mathematics_0d1_82_86_2_documentation_0i_2mathematics_autotoc_md1672}

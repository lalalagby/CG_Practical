 The Input \doxylink{namespace_system}{System} supports multi-\/user management through the \href{../api/UnityEngine.InputSystem.Users.InputUser.html}{\texttt{ {\ttfamily Input\+User}}} class. This comprises both user account management features on platforms that have these capabilities built into them (such as Xbox and PS4), as well as features to manage Device allocations to one or more local users.

\texorpdfstring{$>$}{>}{\bfseries{Note}}\+: The user management API is quite low-\/level in nature. The stock functionality of \doxylink{class_player}{Player} Input Manager component (see \doxysectlink{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_player_input_manager}{Player Input Manager}{0}) provides an easier way to set up user management. The API described here is useful when you want more control over user management.

In the Input \doxylink{namespace_system}{System}, each \href{../api/UnityEngine.InputSystem.Users.InputUser.html}{\texttt{ {\ttfamily Input\+User}}} represents a human interacting with the application. For example, you can have multiple users playing a game together on a single computer or device (local multiplayer), where each user has one or more paired Input Devices.

The \doxysectlink{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_player_input_manager}{{\ttfamily Player\+Input\+Manager}}{0} class uses \href{../api/UnityEngine.InputSystem.Users.InputUser.html}{\texttt{ {\ttfamily Input\+User}}} internally to handle users.

\texorpdfstring{$>$}{>}{\bfseries{Note}}\+: In the editor, all {\ttfamily Input\+User} instances are automatically removed when exiting play mode thus also removing any device pairings. In essence, {\ttfamily Input\+User} is considered a player-\/only API.\hypertarget{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_user_management_autotoc_md1601}{}\doxysection{\texorpdfstring{Device pairing}{Device pairing}}\label{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_user_management_autotoc_md1601}

You can also optionally pass in an existing \href{../api/UnityEngine.InputSystem.Users.InputUser.html}{\texttt{ {\ttfamily Input\+User}}} instance to pair it with the Device, if you don\textquotesingle{}t want to create a new user instance.

When you activate a Control Scheme, the Input \doxylink{namespace_system}{System} automatically switches the active Binding mask for the user\textquotesingle{}s Actions to that Control Scheme.\hypertarget{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_user_management_autotoc_md1603}{}\doxysubsection{\texorpdfstring{Loss of Device}{Loss of Device}}\label{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_user_management_autotoc_md1603}
If paired Input Devices disconnect during the session, the system notifies the \href{../api/UnityEngine.InputSystem.Users.InputUser.html}{\texttt{ {\ttfamily Input\+User}}} class. It still keeps track of the Device, and automatically re-\/pairs the Device if it becomes available again.

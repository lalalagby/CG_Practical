\doxysubsection{Detailed Description}
An asset that contains action maps and control schemes. 

Input\+Action\+Assets can be created in code but are usually stored in JSON format on disk with the "{}.\+inputactions"{} extension. \doxylink{namespace_unity}{Unity} imports them with a custom importer.

To create an \doxylink{class_unity_engine_1_1_input_system_1_1_input_action_asset}{Input\+Action\+Asset} in code, use the {\ttfamily Singleton} API and populate the asset with the methods found in Input\+Action\+Setup\+Extensions. Alternatively, you can use \doxylink{class_unity_engine_1_1_input_system_1_1_input_action_asset_a85dffa2d7f4e53a8a2b15e1ac53d4411}{From\+Json} to load an \doxylink{class_unity_engine_1_1_input_system_1_1_input_action_asset}{Input\+Action\+Asset} directly from a string in JSON format.

\DoxyCodeLine{\textcolor{comment}{//\ Create\ and\ configure\ an\ asset\ in\ code.}}
\DoxyCodeLine{var\ asset1\ =\ ScriptableObject.CreateInstance<InputActionAsset>();}
\DoxyCodeLine{var\ actionMap1\ =\ asset1.AddActionMap(\textcolor{stringliteral}{"{}map1"{}});}
\DoxyCodeLine{action1Map.AddAction(\textcolor{stringliteral}{"{}action1"{}},\ binding:\ \textcolor{stringliteral}{"{}<Keyboard>/space"{}});}

\end{DoxyCode}
 

If you use the API to modify an \doxylink{class_unity_engine_1_1_input_system_1_1_input_action_asset}{Input\+Action\+Asset} while in Play mode, it does not survive the transition back to Edit Mode. \doxylink{namespace_unity}{Unity} tracks and reloads modified assets from disk when exiting Play mode. This is done so that you can realistically test the input related functionality of your application i.\+e. control rebinding etc, without inadvertently changing the input asset.

Each asset can contain arbitrary many action maps that you can enable and disable individually (see \doxylink{class_unity_engine_1_1_input_system_1_1_input_action_map_a1070fcb5a4631ca6c0212fb28a19644f}{Input\+Action\+Map.\+Enable} and \doxylink{class_unity_engine_1_1_input_system_1_1_input_action_map_a215d9741eaf7205b3ad4ac865efae047}{Input\+Action\+Map.\+Disable}) or in bulk (see \doxylink{class_unity_engine_1_1_input_system_1_1_input_action_asset_ae31d59300c207c696f79acb9622da917}{Enable} and \doxylink{class_unity_engine_1_1_input_system_1_1_input_action_asset_a55579dc0d3f6b512cf46518e900fe118}{Disable}). The name of each action map must be unique. The list of action maps can be queried from \doxylink{class_unity_engine_1_1_input_system_1_1_input_action_asset_ac4ec8cdd2eb8b570b9d598967c942c6b}{action\+Maps}.

Input\+Action\+Assets can only define \doxylink{struct_unity_engine_1_1_input_system_1_1_input_control_scheme}{Input\+Control\+Scheme}s. They can be added to an asset with Input\+Action\+Setup\+Extensions.\+Add\+Control\+Scheme(\+Input\+Action\+Asset,string) and can be queried from \doxylink{class_unity_engine_1_1_input_system_1_1_input_action_asset_af40f839c42f5119c08e408ce01deec78}{control\+Schemes}.

Be aware that input action assets do not separate between static (configuration) data and dynamic (instance) data. For audio, for example, {\ttfamily Audio\+Clip} represents the static, shared data portion of audio playback whereas {\ttfamily Audio\+Source"{}$<$/c$>$ represents the dynamic, per-\/instance audio playback portion (referencing the clip through $<$c$>$\+Audio\+Source.\+clip$<$/c$>$).  \+For input, such a split is less beneficial as the same input is generally not exercised multiple times in parallel. Keeping both static and dynamic data together simplifies using the system.  \+However, there are scenarios where you indeed want to take the same input action and exercise it multiple times in parallel. A prominent example of such a use case is local multiplayer where each player gets the same set of actions but is controlling them with a different device (or devices) each. This is easily achieved by simply using $<$c$>$\+Unity\+Engine.\+Object.\+Instantiate$<$/c$>$ to instantiate the input action asset multiple times. $<$see cref="{}\doxylink{class_unity_engine_1_1_input_system_1_1_player_input}{Player\+Input}"{}/\texorpdfstring{$>$}{>} will automatically do so in its internals.}

{\ttfamily Note also that all action maps in an asset share binding state. This means that if one map in an asset has to resolve its bindings, all maps in the asset have to. }

In the latter case, all maps are searched in order and the first action that has the given name in any of the maps is returned. Note that name comparisons are case-\/insensitive.\\
\hline
\end{DoxyParams}
Alternatively, the given string can be a GUID as given by \doxylink{class_unity_engine_1_1_input_system_1_1_input_action_a20c55a69282fe033ba0ca4ce706e4a68}{Input\+Action.\+id}.


\begin{DoxyParams}{Parameters}
{\em throw\+If\+Not\+Found} & If {\ttfamily true}, instead of returning {\ttfamily null} when the action cannot be found, throw {\ttfamily Argument\+Exception}.\\
\hline
\end{DoxyParams}
\begin{DoxyReturn}{Returns}
The action with the corresponding name or {\ttfamily null} if no matching action could be found.
\end{DoxyReturn}


Note that no lookup structures are used internally to speed the operation up. Instead, the search is done linearly. For repeated access of an action, it is thus generally best to look up actions once ahead of time and cache the result.

If multiple actions have the same name and {\itshape action\+Name\+Or\+Id}  is not an ID and not an action name qualified by a map name (that is, in the form of {\ttfamily "{}map\+Name/action\+Name"{}}), the action that is returned will be from the first map in \doxylink{class_unity_engine_1_1_input_system_1_1_input_action_asset_ac4ec8cdd2eb8b570b9d598967c942c6b}{action\+Maps} that has an action with the given name. An exception is if, of the multiple actions with the same name, some are enabled and some are disabled. In this case, the first action that is enabled is returned.

\mbox{\hyperlink{struct_unity_engine_1_1_input_system_1_1_input_binding}{Input\+Binding}}? Unity\+Engine.\+Input\+System.\+Input\+Action\+Asset.\+binding\+Mask\hspace{0.3cm}{\ttfamily [get]}, {\ttfamily [set]}}



Binding mask to apply to all action maps and actions in the asset. 

Optional mask that determines which bindings in the asset to enable.

Binding masks can be applied at three different levels\+: for an entire asset through this property, for a specific map through \doxylink{class_unity_engine_1_1_input_system_1_1_input_action_map_af68ee392460d425649d96935d8f7345d}{Input\+Action\+Map.\+binding\+Mask}, and for single actions through \doxylink{class_unity_engine_1_1_input_system_1_1_input_action_a41caa3a44bc876781c5b1bcb04622cd2}{Input\+Action.\+binding\+Mask}. By default, none of the masks will be set (i.\+e. they will be {\ttfamily null}).

When an action is enabled, all the binding masks that apply to it are taken into account. Specifically, this means that any given binding on the action will be enabled only if it matches the mask applied to the asset, the mask applied to the map that contains the action, and the mask applied to the action itself. All the masks are individually optional.

Masks are matched against bindings using \doxylink{struct_unity_engine_1_1_input_system_1_1_input_binding_aa69fd73f446257c84723c95ad8fbcbe8}{Input\+Binding.\+Matches}.

Note that if you modify the masks applicable to an action while it is enabled, the action\textquotesingle{}s \doxylink{class_unity_engine_1_1_input_system_1_1_input_action_a8e3f5d3329be13ca8474314137045af4}{Input\+Action.\+controls} will get updated immediately to respect the mask. To avoid repeated binding resolution, it is most efficient to apply binding masks before enabling actions.

Binding masks are non-\/destructive. All the bindings on the action are left in place. Setting a mask will not affect the value of the \doxylink{class_unity_engine_1_1_input_system_1_1_input_action_a44fdaf3ff9ffc22609a9df942372d9bb}{Input\+Action.\+bindings} and \doxylink{class_unity_engine_1_1_input_system_1_1_input_action_map_ae98277c3697a798a28f8b16e187217ff}{Input\+Action\+Map.\+bindings} properties.

Unity\+Engine.\+Input\+System.\+Input\+Action\+Asset.\+devices\hspace{0.3cm}{\ttfamily [get]}, {\ttfamily [set]}}



Set of devices that bindings in the asset can bind to. 

Optional set of devices to use by bindings in the asset.

By default (with this property being {\ttfamily null}), bindings will bind to any of the controls available through Input\+System.\+devices, i.\+e. controls from all devices in the system will be used.

By setting this property, binding resolution can instead be restricted to just specific devices. This restriction can either be applied to an entire asset using this property or to specific action maps by using \doxylink{class_unity_engine_1_1_input_system_1_1_input_action_map_ab32ae5eb39f8c39dcebccb5cc1bb99e9}{Input\+Action\+Map.\+devices}. Note that if both this property and \doxylink{class_unity_engine_1_1_input_system_1_1_input_action_map_ab32ae5eb39f8c39dcebccb5cc1bb99e9}{Input\+Action\+Map.\+devices} is set for a specific action map, the list of devices on the action map will take precedence and the list on the asset will be ignored for bindings in that action map.

{\em action\+Name\+Or\+Id} & Name of the action as either a "{}map/action"{} combination (e.\+g. "{}gameplay/fire"{}) or a simple name. In the former case, the name is split at the \textquotesingle{}/\textquotesingle{} slash and the first part is used to find a map with that name and the second part is used to find an action with that name inside the map. In the latter case, all maps are searched in order and the first action that has the given name in any of the maps is returned. Note that name comparisons are case-\/insensitive.\\

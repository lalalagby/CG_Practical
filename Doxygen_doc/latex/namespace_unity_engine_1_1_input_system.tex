Actions can be triggered in steps. For example, a \doxylink{}{\textquotesingle{}slow tap\textquotesingle{}} will put an action into \doxylink{namespace_unity_engine_1_1_input_system_aecc012480a29c4b720a87b8465f34182a8428552d86c0d262a542a528af490afa}{Started} phase when a button the action is bound to is pressed. Without any interactions, an action will go straight from \doxylink{namespace_unity_engine_1_1_input_system_aecc012480a29c4b720a87b8465f34182a5706de961fb376d701be6e7762d8b09c}{Waiting} into \doxylink{namespace_unity_engine_1_1_input_system_aecc012480a29c4b720a87b8465f34182a916b54ec8bd2d9be03994b6486310ed3}{Performed} and back into \doxylink{namespace_unity_engine_1_1_input_system_aecc012480a29c4b720a87b8465f34182a5706de961fb376d701be6e7762d8b09c}{Waiting} whenever an associated control changes value.

\doxylink{namespace_unity_engine_1_1_input_system_aecc012480a29c4b720a87b8465f34182a8428552d86c0d262a542a528af490afa}{Started} can be useful for \doxylink{namespace_unity_engine_1_1_u_i}{UI} feedback. Determines the behavior with which an \doxylink{class_unity_engine_1_1_input_system_1_1_input_action}{Input\+Action} triggers. 

While all actions essentially function the same way, there are differences in how an action will react to changes in values on the controls it is bound to.

The most straightforward type of behavior is \doxylink{namespace_unity_engine_1_1_input_system_a0d0d044aae0aef15123e2f31f11ea94bae1ff6a7360e99cb46db6bb5d3ff8e2eb}{Pass\+Through} which does not expect any kind of value change pattern but simply triggers the action on every single value change. A pass-\/through action will not use \doxylink{class_unity_engine_1_1_input_system_1_1_input_action_a5fecf2acd129fa0922452ecf32527cdd}{Input\+Action.\+started} or \doxylink{class_unity_engine_1_1_input_system_1_1_input_action_a905374d6d7af6dc57a54acfa22018976}{Input\+Action.\+canceled} except on bindings that have an interaction added to them. Pass-\/through actions are most useful for sourcing input from arbitrary many controls and simply piping all input through without much processing on the side of the action.

Note that pass-\/through actions do not perform any kind of disambiguation of input which makes them great for just forwarding input from any connected controls but makes them a poor choice when only one input should be generated from however many connected controls there are. For more details, see \href{../manual/ActionBindings.html\#conflicting-inputs}{\texttt{ here}}.

The other two behavior types are \doxylink{namespace_unity_engine_1_1_input_system_a0d0d044aae0aef15123e2f31f11ea94ba87b7760f14fbff78d8819291f36ab9a0}{Button} and \doxylink{namespace_unity_engine_1_1_input_system_a0d0d044aae0aef15123e2f31f11ea94ba689202409e48743b914713f96d93947c}{Value}.

A \doxylink{namespace_unity_engine_1_1_input_system_a0d0d044aae0aef15123e2f31f11ea94ba689202409e48743b914713f96d93947c}{Value} action starts (\doxylink{class_unity_engine_1_1_input_system_1_1_input_action_a5fecf2acd129fa0922452ecf32527cdd}{Input\+Action.\+started}) as soon as its input moves away from its default value. After that it immediately performs (\doxylink{class_unity_engine_1_1_input_system_1_1_input_action_a11c046d8fb911d300e30e67827222c1f}{Input\+Action.\+performed}) and every time the input changes value it performs again except if the input moves back to the default value -- in which case the action cancels (\doxylink{class_unity_engine_1_1_input_system_1_1_input_action_a905374d6d7af6dc57a54acfa22018976}{Input\+Action.\+canceled}).

Also, unlike both \doxylink{namespace_unity_engine_1_1_input_system_a0d0d044aae0aef15123e2f31f11ea94ba87b7760f14fbff78d8819291f36ab9a0}{Button} and \doxylink{namespace_unity_engine_1_1_input_system_a0d0d044aae0aef15123e2f31f11ea94bae1ff6a7360e99cb46db6bb5d3ff8e2eb}{Pass\+Through} actions, \doxylink{namespace_unity_engine_1_1_input_system_a0d0d044aae0aef15123e2f31f11ea94ba689202409e48743b914713f96d93947c}{Value} actions perform what\textquotesingle{}s called "{}initial state check"{} on the first input update after the action was enabled. What this does is check controls bound to the action and if they are already actuated (that is, at non-\/default value), the action will immediately be started and performed. What this means in practice is that when a value action is bound to, say, the left stick on a gamepad and the stick is already moved out of its resting position, then the action will immediately trigger instead of first requiring the stick to be moved slightly.

\doxylink{namespace_unity_engine_1_1_input_system_a0d0d044aae0aef15123e2f31f11ea94ba87b7760f14fbff78d8819291f36ab9a0}{Button} and \doxylink{namespace_unity_engine_1_1_input_system_a0d0d044aae0aef15123e2f31f11ea94bae1ff6a7360e99cb46db6bb5d3ff8e2eb}{Pass\+Through} actions, on the other hand, perform no such initial state check. For buttons, for example, this means that if a button is already pressed when an action is enabled, it first has to be released and then pressed again for the action to be triggered.

A \doxylink{namespace_unity_engine_1_1_input_system_a0d0d044aae0aef15123e2f31f11ea94ba87b7760f14fbff78d8819291f36ab9a0}{Button} action essentially operates like a \doxylink{namespace_unity_engine_1_1_input_system_a0d0d044aae0aef15123e2f31f11ea94ba689202409e48743b914713f96d93947c}{Value} action except that it does not perform an initial state check.

One final noteworthy difference of both \doxylink{namespace_unity_engine_1_1_input_system_a0d0d044aae0aef15123e2f31f11ea94ba87b7760f14fbff78d8819291f36ab9a0}{Button} and \doxylink{namespace_unity_engine_1_1_input_system_a0d0d044aae0aef15123e2f31f11ea94ba689202409e48743b914713f96d93947c}{Value} compared to \doxylink{namespace_unity_engine_1_1_input_system_a0d0d044aae0aef15123e2f31f11ea94bae1ff6a7360e99cb46db6bb5d3ff8e2eb}{Pass\+Through} is that both of them perform what is referred to as "{}disambiguation"{} when multiple actions are bound to the control. \doxylink{namespace_unity_engine_1_1_input_system_a0d0d044aae0aef15123e2f31f11ea94bae1ff6a7360e99cb46db6bb5d3ff8e2eb}{Pass\+Through} does not care how many controls are bound to the action -- it simply passes every input through as is, no matter where it comes from.

\doxylink{namespace_unity_engine_1_1_input_system_a0d0d044aae0aef15123e2f31f11ea94ba87b7760f14fbff78d8819291f36ab9a0}{Button} and \doxylink{namespace_unity_engine_1_1_input_system_a0d0d044aae0aef15123e2f31f11ea94ba689202409e48743b914713f96d93947c}{Value}, on the other hand, will treat input differently if it is coming from several sources at the same time. Note that this can only happen when there are multiple controls bound to a single actions -- either by a single binding resolving to more than one control (e.\+g. {\ttfamily "{}\texorpdfstring{$\ast$}{*}/\{\+Primary\+Action\}"{}}) or by multiple bindings all targeting the same action and being active at the same time. If only a single control is bound to an action, then the disambiguation code is automatically bypassed.

Disambiguation works the following way\+: when an action has not yet been started, it will react to the first input that has a non-\/default value. Once it receives such an input, it will start tracking the source of that input. While the action is in-\/progress, if it receives input from a source other than the control it is currently tracking, it will check whether the input has a greater magnitude (see \doxylink{class_unity_engine_1_1_input_system_1_1_input_control_a5c3a30406c52cd811f35931b15aa1d5c}{Input\+Control.\+Evaluate\+Magnitude()}) than the control the action is already tracking. If so, the action will switch from its current control to the control with the stronger input.

Note that this process does also works in reverse. When the control currently driving the action lowers its value below that of another control that is also actuated and bound to the action, the action will switch to that control.

Put simply, a \doxylink{namespace_unity_engine_1_1_input_system_a0d0d044aae0aef15123e2f31f11ea94ba87b7760f14fbff78d8819291f36ab9a0}{Button} or \doxylink{namespace_unity_engine_1_1_input_system_a0d0d044aae0aef15123e2f31f11ea94ba689202409e48743b914713f96d93947c}{Value} action bound to multiple controls will always track the control with the strongest input.

\raisebox{\heightof{T}}[0pt][0pt]{\index{PassThrough@{PassThrough}!UnityEngine.InputSystem@{UnityEngine.InputSystem}}\index{UnityEngine.InputSystem@{UnityEngine.InputSystem}!PassThrough@{PassThrough}}}\Hypertarget{namespace_unity_engine_1_1_input_system_a0d0d044aae0aef15123e2f31f11ea94bae1ff6a7360e99cb46db6bb5d3ff8e2eb}\label{namespace_unity_engine_1_1_input_system_a0d0d044aae0aef15123e2f31f11ea94b} 
Pass\+Through&An action that has no specific type of behavior and instead acts as a simple pass-\/through for any value change on any bound control. In effect, this turns an action from a single value producer into a mere input "{}sink"{}. This is in some ways similar to \doxylink{namespace_unity_engine_1_1_input_system_a0d0d044aae0aef15123e2f31f11ea94ba689202409e48743b914713f96d93947c}{Value}. However, there are two key differences.

For one, the action will not perform any disambiguation when bound to multiple controls concurrently. This means that if, for example, the action is bound to both the left and the right stick on a \doxylink{class_unity_engine_1_1_input_system_1_1_gamepad}{Gamepad}, and the left stick goes to (0.\+5,0.\+5) and the right stick then goes to (0.\+25,0.\+25), the action will perform twice yielding a value of (0.\+5,0.\+5) first and a value of (0.\+25, 0.\+25) next. This is different from \doxylink{namespace_unity_engine_1_1_input_system_a0d0d044aae0aef15123e2f31f11ea94ba689202409e48743b914713f96d93947c}{Value} where upon actuation to (0.\+5,0.\+5), the left stick would get to drive the action and the actuation of the right stick would be ignored as it does not exceed the magnitude of the actuation on the left stick.

The second key difference is that only \doxylink{namespace_unity_engine_1_1_input_system_aecc012480a29c4b720a87b8465f34182a916b54ec8bd2d9be03994b6486310ed3}{Input\+Action\+Phase.\+Performed} is used and will get triggered on every value change regardless of what the value is. This is different from \doxylink{namespace_unity_engine_1_1_input_system_a0d0d044aae0aef15123e2f31f11ea94ba689202409e48743b914713f96d93947c}{Value} where the action will trigger \doxylink{namespace_unity_engine_1_1_input_system_aecc012480a29c4b720a87b8465f34182a8428552d86c0d262a542a528af490afa}{Input\+Action\+Phase.\+Started} when moving away from its default value and will trigger \doxylink{namespace_unity_engine_1_1_input_system_aecc012480a29c4b720a87b8465f34182a0e22fe7d45f8e5632a4abf369b24e29c}{Input\+Action\+Phase.\+Canceled} when going back to the default value.

Note that a pass-\/through action my still get cancelled and thus see \doxylink{class_unity_engine_1_1_input_system_1_1_input_action_a905374d6d7af6dc57a54acfa22018976}{Input\+Action.\+canceled} getting called. This happens when a factor other than input on a device causes an action in progress to be cancelled. An example of this is when an action is disabled (see \doxylink{class_unity_engine_1_1_input_system_1_1_input_action_a301a6f1f456e71eee25363f62861ff93}{Input\+Action.\+Disable}) or when focus is lost (see \doxylink{class_unity_engine_1_1_input_system_1_1_input_settings_a11467cec652b53340f93f7a788dac2d5}{Input\+Settings.\+background\+Behavior}) and a device connection to an action is reset (see Input\+System.\+Reset\+Device).

Also note that for a pass-\/through action, calling \doxylink{class_unity_engine_1_1_input_system_1_1_input_action_a49b189fd9d09cc28efd17c08a17eeb0c}{Input\+Action.\+Read\+Value$<$\+TValue$>$} is often not very useful as it will only return the value of the very last control that fed into the action. For pass-\/through actions, it is usually best to listen to \doxylink{class_unity_engine_1_1_input_system_1_1_input_action_a11c046d8fb911d300e30e67827222c1f}{Input\+Action.\+performed} in order to be notified about every single value change. Where this is not necessary, it is generally better to employ a \doxylink{namespace_unity_engine_1_1_input_system_a0d0d044aae0aef15123e2f31f11ea94ba689202409e48743b914713f96d93947c}{Value} action instead. \\

\doxysubsection{Detailed Description}
Represents a specific user/player interacting with one or more devices and input actions. 

Principally, an \doxylink{struct_unity_engine_1_1_input_system_1_1_users_1_1_input_user}{Input\+User} represents a human interacting with the application. Moreover, at any point each \doxylink{struct_unity_engine_1_1_input_system_1_1_users_1_1_input_user}{Input\+User} represents a human actor distinct from all other Input\+Users in the system.

Each user has one or more paired devices. In general, these devices are unique to each user. However, it is permitted to use \doxylink{struct_unity_engine_1_1_input_system_1_1_users_1_1_input_user_ab98da986aae96dd4fa25da37ea976a52}{Perform\+Pairing\+With\+Device} to pair the same device to multiple users. This can be useful in setups such as split-\/keyboard (e.\+g. one user using left side of keyboard and the other the right one) use or hotseat-\/style gameplay (e.\+g. two players taking turns on the same game controller).

Note that the \doxylink{struct_unity_engine_1_1_input_system_1_1_users_1_1_input_user}{Input\+User} API, like \doxylink{class_unity_engine_1_1_input_system_1_1_input_action}{Input\+Action}) is a play mode-\/only feature. When exiting play mode, all users are automatically removed and all devices automatically unpaired.

REVIEW\+: allow re-\/adding a user through this method?


\begin{DoxyParams}{Parameters}
{\em device} & Device to pair to a user.\\
\hline
{\em user} & Optional parameter. If given, instead of creating a new user to pair the device to, the device is paired to the given user.\\
\hline

By default, a new user is created and {\itshape device}  is added \doxylink{struct_unity_engine_1_1_input_system_1_1_users_1_1_input_user_a6ba8d6b588579cc0eb1151d9d37f6476}{paired\+Devices} of the user and \doxylink{namespace_unity_engine_1_1_input_system_1_1_users_ad375fe131f0c5e8721a77a72304a7fe9a7fa309565d7f6c75a745c7034d34c2c7}{Input\+User\+Change.\+Device\+Paired} is sent on \doxylink{struct_unity_engine_1_1_input_system_1_1_users_1_1_input_user_a8b31657fd8cd741cc3b6d1abb36ea4fb}{on\+Change}.

If a valid user is supplied to {\itshape user} , the device is paired to the given user instead of creating a new user. By default, the device is added to the list of already paired devices for the user. This can be changed by using \doxylink{namespace_unity_engine_1_1_input_system_1_1_users_a8ae99c3196547e15f2baab6435e1d4d3a6cc6914936d05d1ad209e5233327a356}{Input\+User\+Pairing\+Options.\+Unpair\+Current\+Devices\+From\+User} which causes devices currently paired to the user to first be unpaired.

The method will not prevent pairing of the same device to multiple users.

Note that if the user has an associated set of actions (\doxylink{struct_unity_engine_1_1_input_system_1_1_users_1_1_input_user_a82f0a302ab236160ef9de88fda450544}{actions}), the list of devices on the actions (\doxylink{interface_unity_engine_1_1_input_system_1_1_i_input_action_collection_a1f8810f3a53393cc9af59f2a88bc2c99}{IInput\+Action\+Collection.\+devices}) will automatically be updated meaning that the newly paired devices will automatically reflect in the set of devices available to the user\textquotesingle{}s actions. If the user has a control scheme that is currently activated (\doxylink{struct_unity_engine_1_1_input_system_1_1_users_1_1_input_user_a89195d650e19407d9bbd47b221489430}{control\+Scheme}), then \doxylink{struct_unity_engine_1_1_input_system_1_1_users_1_1_input_user_a862f24c7fe5b02ccd372d81d72b459bf}{control\+Scheme\+Match} will also automatically update to reflect the matching of devices to the control scheme\textquotesingle{}s device requirements.

Checking a given event for activity of interest is relatively fast but is still costlier than not doing it all. In case only certain devices are of interest for \doxylink{struct_unity_engine_1_1_input_system_1_1_users_1_1_input_user_a96e233b011c91aaaf43241ed007ff127}{on\+Unpaired\+Device\+Used}, this "{}pre-\/filter"{} can be used to quickly reject entire devices and thus skip looking closer at an event.

The first argument is the \doxylink{class_unity_engine_1_1_input_system_1_1_input_device}{Input\+Device} than an event has been received for. The second argument is the Input\+Event that is being looked at.

A callback should return {\ttfamily true} if it wants \doxylink{struct_unity_engine_1_1_input_system_1_1_users_1_1_input_user_a96e233b011c91aaaf43241ed007ff127}{on\+Unpaired\+Device\+Used} to proceed looking at the event and should return {\ttfamily false} if the event should be skipped.

If multiple callbacks are added to the event, it is enough for any single one callback to return {\ttfamily true} for the event to get looked at.

Event that is triggered when a device is used that is not currently paired to any user. 

A device is considered "{}used"{} when it has magnitude (\doxylink{class_unity_engine_1_1_input_system_1_1_input_control_a5c3a30406c52cd811f35931b15aa1d5c}{Input\+Control.\+Evaluate\+Magnitude()}) greater than zero on a control that is not noisy (\doxylink{class_unity_engine_1_1_input_system_1_1_input_control_afa5a8531d7c9a030e8e484560a59fe6f}{Input\+Control.\+noisy}) and not synthetic (i.\+e. not a control that is "{}made up"{} like \doxylink{class_unity_engine_1_1_input_system_1_1_keyboard_ade8d96be2b5c7890e7389d2e7e8b4350}{Keyboard.\+any\+Key}; \doxylink{class_unity_engine_1_1_input_system_1_1_input_control_a11348388ab9565068c0f1294de909b90}{Input\+Control.\+synthetic}).

Detecting the use of unpaired devices has a non-\/zero cost. While multiple levels of tests are applied to try to cheaply ignore devices that have events sent to them that do not contain user activity, finding out whether a device had real user activity will eventually require going through the device control by control.

To enable detection of the use of unpaired devices, set \doxylink{struct_unity_engine_1_1_input_system_1_1_users_1_1_input_user_aaa38daa9f9509c6bdd565c5978b7cd38}{listen\+For\+Unpaired\+Device\+Activity} to true. It is disabled by default.

The callback is invoked for each non-\/leaf, non-\/synthetic, non-\/noisy control that has been actuated on the device. It being restricted to non-\/leaf controls means that if, say, the stick on a gamepad is actuated in both X and Y direction, you will see two calls\+: one with stick/x and one with stick/y.

The reason that the callback is invoked for each individual control is that pairing often relies on checking for specific kinds of interactions. For example, a pairing callback may listen exclusively for button presses.

Note that whether the use of unpaired devices leads to them getting paired is under the control of the application. If the device should be paired, invoke \doxylink{struct_unity_engine_1_1_input_system_1_1_users_1_1_input_user_ab98da986aae96dd4fa25da37ea976a52}{Perform\+Pairing\+With\+Device} from the callback. If you do so, no further callbacks will get triggered for other controls that may have been actuated in the same event.

Be aware that the callback is fired {\itshape before} input is actually incorporated into the device (it is indirectly triggered from Input\+System.\+on\+Event). This means at the time the callback is run, the state of the given device does not yet have the input that triggered the callback. For this reason, the callback receives a second argument that references the event from which the use of an unpaired device was detected.

What this sequence allows is to make changes to the system before the input is processed. For example, an action that is enabled as part of the callback will subsequently respond to the input that triggered the callback.

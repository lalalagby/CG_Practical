\doxysubsection{Detailed Description}
Project-\/wide input settings. 

Several aspects of the input system can be customized to tailor how the system functions to the specific needs of a project. These settings are collected in this class. There is one global settings object active at any one time. It can be accessed and set through Input\+System.\+settings.

Changing a setting on the object takes effect immediately. It also triggers the Input\+System.\+on\+Settings\+Change callback.

Reset\+And\+Disable\+Non\+Background\+Devices&When the application loses focus, issue a Input\+System.\+Reset\+Device call on every \doxylink{class_unity_engine_1_1_input_system_1_1_input_device}{Input\+Device} that is not marked as \doxylink{class_unity_engine_1_1_input_system_1_1_input_device_acecc008067d46ee9a1db3cf059f6c709}{Input\+Device.\+can\+Run\+In\+Background} and then disable the device (see Input\+System.\+Disable\+Device and \doxylink{class_unity_engine_1_1_input_system_1_1_input_device_a11b982f33c9ccecd0cbf73501e363e8f}{Input\+Device.\+enabled}). Devices that \doxylink{class_unity_engine_1_1_input_system_1_1_input_device_acecc008067d46ee9a1db3cf059f6c709}{Input\+Device.\+can\+Run\+In\+Background} will not be touched and will keep running as is. In effect, this setting will "{}soft-\/reset"{} all devices that cannot receive input while the application does not have focus. That is, it will reset all controls that are not marked as \doxylink{struct_unity_engine_1_1_input_system_1_1_layouts_1_1_input_control_layout_1_1_control_item_a60f4e5350e3075c9044005219f445983}{Input\+Control\+Layout.\+Control\+Item.\+dont\+Reset} to their default state.

When the application comes back into focus, all devices that have been reset and disabled will be re-\/enabled and a synchronization request (see Request\+Sync\+Command) will be sent to each device.

Devices that are added while the application is running in the background are treated like devices that were already present when losing focus. That is, if they cannot run in the background, they will be disabled until focus comes back.

Note that the resets will cancel \doxylink{class_unity_engine_1_1_input_system_1_1_input_action}{Input\+Action}s that are in progress from controls on devices that are being reset. \\

Default button press threshold.

In the input system, each button constitutes a full floating-\/point value. Pure toggle buttons, such as \doxylink{class_unity_engine_1_1_input_system_1_1_gamepad_a55b57fe1d0a61c4fe2e671284ec2ab3b}{Gamepad.\+button\+South} for example, will simply alternate between 0 (not pressed) and 1 (pressed). When used in a context where a clear distinction between pressed and not pressed is required, we need a value beyond which we consider the button pressed.

By setting this property, the default value for this can be configured. If a button has a value equal to or greater than the button press point, it is considered pressed.

The default value is 0.\+5.

Any value will implicitly be clamped to {\ttfamily 0.\+0001f} as allowing a value of 0 would cause all buttons in their default state to already be pressed.

Lowering the button press point will make triggers feel more like hair-\/triggers (akin to using the hair-\/trigger feature on Xbox Elite controllers). However, it may make using the directional buttons (i.\+e. \doxylink{class_unity_engine_1_1_input_system_1_1_controls_1_1_stick_control_a8eead7497ff7ca10a99ae1f843d907d8}{Controls.\+Stick\+Control.\+up} etc) be fickle as solely moving in only one direction with sticks isn\textquotesingle{}t easy. To counteract that, the button press points on the stick buttons can be raised.

Another solution is to simply lower the press points on the triggers specifically.

Default upper limit for deadzones.

"{}\+Deadzones"{} refer to limits established for the range of values accepted as input on a control. If the value for the control falls outside the range, i.\+e. below the given minimum or above the given maximum, the value is clamped to the respective limit.

This property configures the default upper bound of the value range.

Note that deadzones will by default re-\/normalize values after clamping. This means that inputs at the lower and upper end are dropped and that the range in-\/between is re-\/normalized to \mbox{[}0..1\mbox{]}.

Note that deadzones preserve the sign of inputs. This means that both the upper and the lower deadzone bound extend to both the positive and the negative range. For example, a deadzone max of 0.\+95 will clamp values of \texorpdfstring{$>$}{>}0.\+95 and \texorpdfstring{$<$}{<}-\/0.\+95.

The most common example of where deadzones are used are the sticks on gamepads, i.\+e. \doxylink{class_unity_engine_1_1_input_system_1_1_gamepad_a9f608a9e12af3301fb2d704e072ea024}{Gamepad.\+left\+Stick} and \doxylink{class_unity_engine_1_1_input_system_1_1_gamepad_a24a1a82fd0aab986f8e7e141dab4035f}{Gamepad.\+right\+Stick}. Sticks will usually be wobbly to some extent (just how wobbly varies greatly between different types of controllers -- which means that often deadzones need to be configured on a per-\/device type basis). Using deadzones, stick motion at the extreme ends of the spectrum can be filtered out and noise in these areas can effectively be eliminated this way.

The default value for this property is 0.\+925.

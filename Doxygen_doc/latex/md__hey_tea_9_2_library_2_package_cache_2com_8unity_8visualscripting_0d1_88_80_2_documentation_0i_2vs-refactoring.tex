 Visual Scripting creates nodes from methods, fields, and properties from C\# script in your project. Visual Scripting creates these nodes after you \doxysectlink{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8visualscripting_0d1_88_80_2_documentation_0i_2vs-configuration}{regenerate your Node Library}{0} and \doxysectlink{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8visualscripting_0d1_88_80_2_documentation_0i_2vs-add-remove-type-options}{add any relevant types to your Type Options}{0}.

For example, Visual Scripting created the following Take Damage node from a custom C\# script that defines the {\ttfamily \doxylink{class_player}{Player}} class.



Visual Scripting generated the node with the following code, which creates a {\ttfamily \doxylink{class_player}{Player}} class with a {\ttfamily Take\+Damage} member.

If you change the name of the {\ttfamily Take\+Damage} member in the C\# script, Visual Scripting displays an error in Script Graphs that use the Take Damage node.

\texorpdfstring{$<$}{<}img src="{}images/vs-\/refactoring-\/script-\/node-\/error-\/example.\+png"{} alt="{}\+An image of the Graph window, that displays the Take Damage node and the Graph Inspector. To rename a member, type, class, struct, enum, or other API element that a Visual Scripting node uses in a project, add the {\ttfamily \mbox{[}Renamed\+From\mbox{]}} attribute to the relevant API element in the script file. To avoid issues with \doxylink{namespace_unity}{Unity}\textquotesingle{}s serialization, the {\ttfamily \mbox{[}Renamed\+From\mbox{]}} attribute tells Visual Scripting that an API or one of its elements has been renamed.

An Interaction represents a specific input pattern. For example, a hold is an Interaction that requires a Control to be held for at least a minimum amount of time.

Interactions drive responses on Actions. You can place them on individual Bindings or an Action as a whole, in which case they apply to every Binding on the Action. At runtime, when a particular interaction completes, this triggers the Action.

\hypertarget{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_interactions_autotoc_md1403}{}\doxysection{\texorpdfstring{Operation}{Operation}}\label{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_interactions_autotoc_md1403}
An Interaction has a set of distinct phases it can go through in response to receiving input.

Not every Interaction triggers every phase, and the pattern in which specific Interactions trigger phases depends on the Interaction type.

While {\ttfamily Performed} is typically the phase that triggers the actual response to an Interaction, {\ttfamily Started} and {\ttfamily Canceled} can be useful for providing UI feedback while the Interaction is in progress. For example, when a hold is {\ttfamily Started}, the app can display a progress bar that fills up until the hold time has been reached. If, however, the hold is {\ttfamily Canceled} before it completes, the app can reset the progress bar to the beginning.

\hypertarget{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_interactions_autotoc_md1404}{}\doxysubsection{\texorpdfstring{Multiple Controls on an Action}{Multiple Controls on an Action}}\label{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_interactions_autotoc_md1404}
If you have multiple Controls bound to a Binding or an Action which has an Interaction, then the Input \doxylink{namespace_system}{System} first applies the \href{ActionBindings.md\#conflicting-inputs}{\texttt{ Control conflict resolution}} logic to get a single value for the Action, which it then feeds to the Interaction logic. Any of the bound Controls can perform the Interaction.\hypertarget{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_interactions_autotoc_md1405}{}\doxysubsection{\texorpdfstring{Multiple Interactions on a Binding}{Multiple Interactions on a Binding}}\label{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_interactions_autotoc_md1405}
If multiple Interactions are present on a single Binding or Action, then the Input \doxylink{namespace_system}{System} checks the Interactions in the order they are present on the Binding. The code example above illustrates this example. The Binding on the {\ttfamily fire\+Action} Action has two Interactions\+: {\ttfamily With\+Interactions("{}tap;slow\+Tap"{})}. The tap Interaction gets a first chance at interpreting the input from the Action. If the button is pressed, the Action calls the {\ttfamily Started} callback on the tap Interaction. If the user keeps holding the button, the tap Interaction times out, and the Action calls the \href{../api/UnityEngine.InputSystem.InputAction.html\#UnityEngine_InputSystem_InputAction_canceled}{\texttt{ {\ttfamily Canceled}}} callback for the tap Interaction and starts processing the slow tap Interaction (which now receives a {\ttfamily Started} callback).

At any one time, only one Interaction can be "{}driving"{} the action (that is, it gets to determine the action\textquotesingle{}s current \href{../api/UnityEngine.InputSystem.InputAction.html\#UnityEngine_InputSystem_InputAction_phase}{\texttt{ {\ttfamily phase}}}). If an Interaction higher up in the stack cancels, Interactions lower down in the stack can take over.\hypertarget{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_interactions_autotoc_md1406}{}\doxysubsection{\texorpdfstring{Timeouts}{Timeouts}}\label{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_interactions_autotoc_md1406}
Interactions might need to wait a certain time for a specific input to occur or to not occur. An example of this is the Hold interaction which, after a button is pressed, has to wait for a set \href{../api/UnityEngine.InputSystem.Interactions.HoldInteraction.html\#UnityEngine_InputSystem_Interactions_HoldInteraction_duration}{\texttt{ duration}} until the "{}hold"{} is complete. To do this, an interaction installs a timeout using \href{../api/UnityEngine.InputSystem.InputInteractionContext.html\#UnityEngine_InputSystem_InputInteractionContext_SetTimeout_System_Single_}{\texttt{ {\ttfamily Set\+Timeout}}}.

It can be useful to know how much of a timeout is left for an interaction to complete. For example, you might want to display a bar in the UI that is charging up while the interaction is waiting to complete. To query the percentage to which a timeout has completed, use \href{../api/UnityEngine.InputSystem.InputAction.html\#UnityEngine_InputSystem_InputAction_GetTimeoutCompletionPercentage_}{\texttt{ {\ttfamily Get\+Timeout\+Completion\+Percentage}}}.


\begin{DoxyCode}{0}
\DoxyCodeLine{\textcolor{comment}{//\ Returns\ a\ value\ between\ 0\ (inclusive)\ and\ 1\ (inclusive).}}
\DoxyCodeLine{var\ warpActionCompletion\ =\ playerInput.actions[\textcolor{stringliteral}{"{}warp"{}}].GetTimeoutCompletionPercentage();}

\end{DoxyCode}


Note that each Interaction can have its own separate timeout (but only a single one at any one time). If multiple interactions are in effect, then \href{../api/UnityEngine.InputSystem.InputAction.html\#UnityEngine_InputSystem_InputAction_GetTimeoutCompletionPercentage_}{\texttt{ {\ttfamily Get\+Timeout\+Completion\+Percentage}}} will only use the timeout of the one interaction that is currently driving the action.

Some Interactions might involve multiple timeouts in succession. In this case, knowing only the completion of the currently running timeout (if any) is often not useful. An example is \href{../api/UnityEngine.InputSystem.Interactions.MultiTapInteraction.html}{\texttt{ {\ttfamily Multi\+Tap\+Interaction}}}, which involves a timeout on each individual tap, as well as a timeout in-\/between taps. The Interaction is complete only after a full tap sequence has been performed.

An Interaction can use \href{../api/UnityEngine.InputSystem.InputInteractionContext.html\#UnityEngine_InputSystem_InputInteractionContext_SetTotalTimeoutCompletionTime_System_Single_}{\texttt{ {\ttfamily Set\+Total\+Timeout\+Completion\+Time}}} to inform the Input \doxylink{namespace_system}{System} of the total time it will run timeouts for.\hypertarget{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_interactions_autotoc_md1407}{}\doxysection{\texorpdfstring{Using Interactions}{Using Interactions}}\label{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_interactions_autotoc_md1407}
You can install Interactions on \doxysectlink{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_action_bindings}{Bindings}{0} or \doxysectlink{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_actions}{Actions}{0}.\hypertarget{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_interactions_autotoc_md1408}{}\doxysubsection{\texorpdfstring{Interactions applied to Bindings}{Interactions applied to Bindings}}\label{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_interactions_autotoc_md1408}
When you create Bindings for your \doxysectlink{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_actions}{Actions}{0}, you can choose to add Interactions to the Bindings.

If you\textquotesingle{}re using \doxysectlink{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_action_assets}{Input Action Assets}{0}, you can add any Interaction to your Bindings in the Input Action editor. Once you \href{ActionAssets.md\#editing-bindings}{\texttt{ created some Bindings}}, select the Binding you want to add Interactions to, so that the right pane of the window shows the properties for that Binding. Next, click on the plus icon on the {\bfseries{Interactions}} foldout to open a list of all available Interactions types. Choose an Interaction type to add an Interaction instance of that type. The Interaction now appears in the {\bfseries{Interactions}} foldout. If the Interaction has any parameters, you can now edit them here as well\+:



To remove an Interaction, click the minus button next to it. To change the order of Interactions, click the up and down arrows.

If you create your Bindings in code, you can add Interactions like this\+:

\DoxyCodeLine{var\ Action\ =\ \textcolor{keyword}{new}\ InputAction();}
\DoxyCodeLine{action.AddBinding(\textcolor{stringliteral}{"{}<Gamepad>/leftStick"{}})}
\DoxyCodeLine{\ \ \ \ .WithInteractions(\textcolor{stringliteral}{"{}tap(duration=0.8)"{}});}

\end{DoxyCode}
\hypertarget{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_interactions_autotoc_md1409}{}\doxysubsection{\texorpdfstring{Interactions applied to Actions}{Interactions applied to Actions}}\label{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_interactions_autotoc_md1409}
Applying Interactions directly to an Action is equivalent to applying them to all Bindings for the Action. It is thus more or less a shortcut that avoids manually adding the same Interaction(s) to each of the Bindings.

If Interactions are applied {\bfseries{both}} to an Action and to its Bindings, then the effect is the same as if the Action\textquotesingle{}s Interactions are {\itshape appended} to the list of Interactions on each of the Bindings. This means that the Binding\textquotesingle{}s Interactions are applied {\itshape first}, and then the Action\textquotesingle{}s Interactions are applied {\itshape after}.

You can add and edit Interactions on Actions in the \doxysectlink{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_action_assets}{Input Action Assets}{0} editor window the same way as you would do for Bindings\+: select an Action to Edit, then add the Interactions in the right window pane.

If you create your Actions in code, you can add Interactions like this\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{var\ Action\ =\ \textcolor{keyword}{new}\ InputAction(Interactions:\ \textcolor{stringliteral}{"{}tap(duration=0.8)"{}});}

\end{DoxyCode}
\hypertarget{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_interactions_autotoc_md1410}{}\doxysection{\texorpdfstring{Predefined Interactions}{Predefined Interactions}}\label{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_interactions_autotoc_md1410}
The Input \doxylink{namespace_system}{System} package comes with a set of basic Interactions you can use. If an Action has no Interactions set, the system uses its default Interaction.

\texorpdfstring{$>$}{>}{\bfseries{Note}}\+: The built-\/in Interactions operate on Control actuation and don\textquotesingle{}t use Control values directly. The Input \doxylink{namespace_system}{System} evaluates the {\ttfamily press\+Point} parameters against the magnitude of the Control actuation. This means you can use these Interactions on any Control which has a magnitude, such as sticks, and not just on buttons.

The following diagram shows the behavior of the built-\/in Interactions for a simple button press.

\hypertarget{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_interactions_autotoc_md1411}{}\doxysubsection{\texorpdfstring{Default Interaction}{Default Interaction}}\label{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_interactions_autotoc_md1411}
If you haven\textquotesingle{}t specifically added an Interaction to a Binding or its Action, the default Interaction applies to the Binding.

\href{Actions.md\#value}{\texttt{ {\ttfamily Value}}} type Actions have the following behavior\+:


\begin{DoxyEnumerate}
\item As soon as a bound Control becomes \href{Controls.md\#control-actuation}{\texttt{ actuated}}, the Action goes from {\ttfamily Waiting} to {\ttfamily Started}, and then immediately to {\ttfamily Performed} and back to {\ttfamily Started}. One callback occurs on \href{../api/UnityEngine.InputSystem.InputAction.html\#UnityEngine_InputSystem_InputAction_started}{\texttt{ {\ttfamily Input\+Action.\+started}}}, followed by one callback on \href{../api/UnityEngine.InputSystem.InputAction.html\#UnityEngine_InputSystem_InputAction_performed}{\texttt{ {\ttfamily Input\+Action.\+performed}}}.
\item For as long as the bound Control remains actuated, the Action stays in {\ttfamily Started} and triggers {\ttfamily Performed} whenever the value of the Control changes (that is, one call occurs to \href{../api/UnityEngine.InputSystem.InputAction.html\#UnityEngine_InputSystem_InputAction_performed}{\texttt{ {\ttfamily Input\+Action.\+performed}}}).
\item When the bound Control stops being actuated, the Action goes to {\ttfamily Canceled} and then back to {\ttfamily Waiting}. One call occurs to \href{../api/UnityEngine.InputSystem.InputAction.html\#UnityEngine_InputSystem_InputAction_canceled}{\texttt{ {\ttfamily Input\+Action.\+canceled}}}.
\end{DoxyEnumerate}

\href{Actions.md\#button}{\texttt{ {\ttfamily Button}}} type Actions have the following behavior\+:


\begin{DoxyEnumerate}
\item As soon as a bound Control becomes \href{Controls.md\#control-actuation}{\texttt{ actuated}}, the Action goes from {\ttfamily Waiting} to {\ttfamily Started}. One callback occurs on \href{../api/UnityEngine.InputSystem.InputAction.html\#UnityEngine_InputSystem_InputAction_started}{\texttt{ {\ttfamily Input\+Action.\+started}}}.
\item If a Control then reaches or exceeds the button press threshold, the Action goes from {\ttfamily Started} to {\ttfamily Performed}. One callback occurs on \href{../api/UnityEngine.InputSystem.InputAction.html\#UnityEngine_InputSystem_InputAction_performed}{\texttt{ {\ttfamily Input\+Action.\+performed}}}. The default value of the button press threshold is defined in the \href{../api/UnityEngine.InputSystem.InputSettings.html\#UnityEngine_InputSystem_InputSettings_defaultButtonPressPoint}{\texttt{ input settings}}. However, an individual control can \href{../api/UnityEngine.InputSystem.Controls.ButtonControl.html\#UnityEngine_InputSystem_Controls_ButtonControl_pressPoint}{\texttt{ override}} this value.
\item Once the Action has {\ttfamily Performed}, if all Controls then go back to a level of actuation at or below the \href{../api/UnityEngine.InputSystem.InputSettings.html\#UnityEngine_InputSystem_InputSettings_buttonReleaseThreshold}{\texttt{ release threshold}}, the Action goes from {\ttfamily Performed} to {\ttfamily Canceled}. One call occurs to \href{../api/UnityEngine.InputSystem.InputAction.html\#UnityEngine_InputSystem_InputAction_canceled}{\texttt{ {\ttfamily Input\+Action.\+canceled}}}.
\item If the Action never went to {\ttfamily Performed}, it will go to {\ttfamily Canceled} as soon as all Controls are released. One call occurs to \href{../api/UnityEngine.InputSystem.InputAction.html\#UnityEngine_InputSystem_InputAction_canceled}{\texttt{ {\ttfamily Input\+Action.\+canceled}}}.
\end{DoxyEnumerate}

\href{Actions.md\#pass-through}{\texttt{ {\ttfamily Pass\+Through}}} type Actions have a simpler behavior. The Input \doxylink{namespace_system}{System} doesn\textquotesingle{}t try to track bound Controls as a single source of input. Instead, it triggers a {\ttfamily Performed} callback for each value change.

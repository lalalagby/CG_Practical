\doxysubsection{Detailed Description}
Represents a separate player in the game complete with a set of actions exclusive to the player and a set of paired device. 

\doxylink{class_unity_engine_1_1_input_system_1_1_player_input}{Player\+Input} is a high-\/level wrapper around much of the input system\textquotesingle{}s functionality which is meant to help getting set up with the new input system quickly. It takes care of \doxylink{class_unity_engine_1_1_input_system_1_1_input_action}{Input\+Action} bookkeeping and has a custom \doxylink{namespace_unity_engine_1_1_u_i}{UI}(requires the "{}\+Unity UI"{} package) to help setting up input.

The component supports local multiplayer implicitly. Each \doxylink{class_unity_engine_1_1_input_system_1_1_player_input}{Player\+Input} instance represents a distinct user with its own set of devices and actions. To orchestrate player management and facilitate mechanics such as joining by device activity, use \doxylink{class_unity_engine_1_1_input_system_1_1_player_input_manager}{Unity\+Engine.\+Input\+System.\+Player\+Input\+Manager}.

The way \doxylink{class_unity_engine_1_1_input_system_1_1_player_input}{Player\+Input} notifies script code of events is determined by \doxylink{class_unity_engine_1_1_input_system_1_1_player_input_ae3dfce7b0d7142631c816096285ad069}{notification\+Behavior}. By default, this is set to \doxylink{namespace_unity_engine_1_1_input_system_ada6c6990c08ec51f071b15c38295208aac1a6cc628d22efbf4e40b3efbb92289b}{Unity\+Engine.\+Input\+System.\+Player\+Notifications.\+Send\+Messages} which will use Game\+Object.\+Send\+Message(string,object) to send messages to the Game\+Object that \doxylink{class_unity_engine_1_1_input_system_1_1_player_input}{Player\+Input} sits on.

When enabled, \doxylink{class_unity_engine_1_1_input_system_1_1_player_input}{Player\+Input} will create an Input\+User and pair devices to the user which are then specific to the player. The set of devices can be controlled explicitly when instantiating a \doxylink{class_unity_engine_1_1_input_system_1_1_player_input}{Player\+Input} through Instantiate(\+Game\+Object,int,string,int,\+Input\+Device\mbox{[}$\,$\mbox{]}) or \doxylink{class_unity_engine_1_1_input_system_1_1_player_input_a7d9b31c551c52f2b455262115660822c}{Instantiate(\+Game\+Object,int,string,int,\+Input\+Device)}. This also makes it possible to assign the same device to two different players, e.\+g. for split-\/keyboard play.

If no specific devices are given to a \doxylink{class_unity_engine_1_1_input_system_1_1_player_input}{Player\+Input}, the component will look for compatible devices present in the system and pair them to itself automatically. If the \doxylink{class_unity_engine_1_1_input_system_1_1_player_input}{Player\+Input}\textquotesingle{}s \doxylink{class_unity_engine_1_1_input_system_1_1_player_input_ace52ec4d5b8b26e2b18ede34280f6b85}{actions} have control schemes defined for them, \doxylink{class_unity_engine_1_1_input_system_1_1_player_input}{Player\+Input} will look for a control scheme for which all required devices are available and not paired to any other player. It will try \doxylink{class_unity_engine_1_1_input_system_1_1_player_input_ae99dc12003e18af4df8a61d27984ce95}{default\+Control\+Scheme} first (if set), but then fall back to trying all available schemes in order. Once a scheme is found for which all required devices are available, \doxylink{class_unity_engine_1_1_input_system_1_1_player_input}{Player\+Input} will pair those devices to itself and select the given scheme.

If no control schemes are defined, \doxylink{class_unity_engine_1_1_input_system_1_1_player_input}{Player\+Input} will try to bind as many as-\/of-\/yet unpaired devices to itself as it can match to bindings present in the \doxylink{class_unity_engine_1_1_input_system_1_1_player_input_ace52ec4d5b8b26e2b18ede34280f6b85}{actions}. This means that if, for example, there\textquotesingle{}s binding for both keyboard and gamepad and there is one keyboard and two gamepads available when \doxylink{class_unity_engine_1_1_input_system_1_1_player_input}{Player\+Input} is enabled, all three devices will be paired to the player.

Note that when using \doxylink{class_unity_engine_1_1_input_system_1_1_player_input_manager}{Player\+Input\+Manager}, device pairing to players is controlled from the joining logic. In that case, \doxylink{class_unity_engine_1_1_input_system_1_1_player_input}{Player\+Input} will automatically pair the device from which the player joined. If control schemes are present in \doxylink{class_unity_engine_1_1_input_system_1_1_player_input_ace52ec4d5b8b26e2b18ede34280f6b85}{actions}, the first one compatible with that device is chosen. If additional devices are required, these will be paired from the pool of currently unpaired devices.

Device pairings can be changed at any time by either manually controlling pairing through \doxylink{struct_unity_engine_1_1_input_system_1_1_users_1_1_input_user_ab98da986aae96dd4fa25da37ea976a52}{Input\+User.\+Perform\+Pairing\+With\+Device} (and related methods) using a \doxylink{class_unity_engine_1_1_input_system_1_1_player_input}{Player\+Input}\textquotesingle{}s assigned \doxylink{class_unity_engine_1_1_input_system_1_1_player_input_a961ce498ad0d74c6a704a70fbb6cf5a4}{user} or by switching control schemes (e.\+g. using Switch\+Current\+Control\+Scheme(string,\+Input\+Device\mbox{[}$\,$\mbox{]})), if any are present in the \doxylink{class_unity_engine_1_1_input_system_1_1_player_input}{Player\+Input}\textquotesingle{}s \doxylink{class_unity_engine_1_1_input_system_1_1_player_input_ace52ec4d5b8b26e2b18ede34280f6b85}{actions}.

When a player loses a device paired to it (e.\+g. when it is unplugged or loses power), Input\+User will signal Input\+User\+Change.\+Device\+Lost which is also surfaced as a message, \doxylink{class_unity_engine_1_1_input_system_1_1_player_input_a5e530e431f5b249327a0694e5a8ec03c}{device\+Lost\+Event}, or \doxylink{class_unity_engine_1_1_input_system_1_1_player_input_a0c76c2ee1d4b218a1c091d045dc492a4}{on\+Device\+Lost} (depending on \doxylink{class_unity_engine_1_1_input_system_1_1_player_input_ae3dfce7b0d7142631c816096285ad069}{notification\+Behavior}). When a device is reconnected, Input\+User will signal Input\+User\+Change.\+Device\+Regained which also is surfaced as a message, as \doxylink{class_unity_engine_1_1_input_system_1_1_player_input_ac4719881cb90443e0e7f73e8d4737475}{device\+Regained\+Event}, or \doxylink{class_unity_engine_1_1_input_system_1_1_player_input_a288c018707a71c31aa19d642fc308529}{on\+Device\+Regained} (depending on \doxylink{class_unity_engine_1_1_input_system_1_1_player_input_ae3dfce7b0d7142631c816096285ad069}{notification\+Behavior}).

When there is only a single active \doxylink{class_unity_engine_1_1_input_system_1_1_player_input}{Player\+Input} in the game, joining is not enabled (see \doxylink{class_unity_engine_1_1_input_system_1_1_player_input_manager_abfe072c89bf293c1bd7ef1cbb504779e}{Player\+Input\+Manager.\+joining\+Enabled}), and \doxylink{class_unity_engine_1_1_input_system_1_1_player_input_a379ffe3b0d724a34ce249fa1b21e6076}{never\+Auto\+Switch\+Control\+Schemes} is not set to {\ttfamily true}, device pairings for the player will also update automatically based on device usage.

If control schemes are present in \doxylink{class_unity_engine_1_1_input_system_1_1_player_input_ace52ec4d5b8b26e2b18ede34280f6b85}{actions}, then if a device is used (not merely plugged in but rather receives input on a non-\/noisy, non-\/synthetic control) which is compatible with a control scheme other than the currently used one, \doxylink{class_unity_engine_1_1_input_system_1_1_player_input}{Player\+Input} will attempt to switch to that control scheme. Success depends on whether all device requirements for that scheme are met from the set of available devices. If a control scheme happens, Input\+User signals Input\+User\+Change.\+Control\+Scheme\+Changed on \doxylink{struct_unity_engine_1_1_input_system_1_1_users_1_1_input_user_a8b31657fd8cd741cc3b6d1abb36ea4fb}{Input\+User.\+on\+Change}.

If no control schemes are present in \doxylink{class_unity_engine_1_1_input_system_1_1_player_input_ace52ec4d5b8b26e2b18ede34280f6b85}{actions}, \doxylink{class_unity_engine_1_1_input_system_1_1_player_input}{Player\+Input} will automatically pair any newly available device to itself if the given device has any bindings available for it.

Both behaviors described in the previous two paragraphs are automatically disabled if more than one \doxylink{class_unity_engine_1_1_input_system_1_1_player_input}{Player\+Input} is active.

Input actions associated with the player. 

Asset holding the player\textquotesingle{}s input actions.

Note that every player will maintain a unique copy of the given actions such that each player receives an identical copy. When assigning the same actions to multiple players, the first player will use the given actions as is but any subsequent player will make a copy of the actions using Object.\+Instantiate(\+Object).

The asset may contain an arbitrary number of action maps. By setting \doxylink{class_unity_engine_1_1_input_system_1_1_player_input_aaef443cd03b8725076e16bbf88627322}{default\+Action\+Map}, one of them can be selected to enabled automatically when \doxylink{class_unity_engine_1_1_input_system_1_1_player_input}{Player\+Input} is enabled. If no default action map is selected, none of the action maps will be enabled by \doxylink{class_unity_engine_1_1_input_system_1_1_player_input}{Player\+Input} itself. Use Switch\+Current\+Action\+Map or just call \doxylink{class_unity_engine_1_1_input_system_1_1_input_action_map_a1070fcb5a4631ca6c0212fb28a19644f}{Input\+Action\+Map.\+Enable} directly to enable a specific map.

Notifications will be sent for all actions in the asset, not just for those in the first action map. This means that if additional maps are manually enabled and disabled, notifications will be sent for their actions as they receive input.

If true, do not automatically switch control schemes even when there is only a single player. By default, this property is false. 

If true, do not switch control schemes when other devices are used.

By default, when there is only a single \doxylink{class_unity_engine_1_1_input_system_1_1_player_input}{Player\+Input} enabled, we assume that the game is in single-\/player mode and that the player should be able to freely switch between the control schemes supported by the game. For example, if the player is currently using mouse and keyboard, but is then switching to a gamepad, \doxylink{class_unity_engine_1_1_input_system_1_1_player_input}{Player\+Input} should automatically switch to the control scheme for gamepads, if present.

When there is more than one \doxylink{class_unity_engine_1_1_input_system_1_1_player_input}{Player\+Input} or when joining is enabled \doxylink{class_unity_engine_1_1_input_system_1_1_player_input_manager}{Player\+Input\+Manager}, this behavior is automatically turned off as we wouldn\textquotesingle{}t know which player is switching if a currently unpaired device is used.

By setting this property to true, auto-\/switching of control schemes is forcibly turned off and will thus not be performed even if there is only a single \doxylink{class_unity_engine_1_1_input_system_1_1_player_input}{Player\+Input} in the game.

Note that you can still switch control schemes manually using Switch\+Current\+Control\+Scheme(string,\+Input\+Device\mbox{[}$\,$\mbox{]}).

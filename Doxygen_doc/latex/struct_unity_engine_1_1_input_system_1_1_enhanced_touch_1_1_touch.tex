\doxysubsection{Detailed Description}
A high-\/level representation of a touch which automatically keeps track of a touch over time. 

This API obsoletes the need for manually keeping tracking of touch IDs (\doxylink{class_unity_engine_1_1_input_system_1_1_controls_1_1_touch_control_ace0129fe132de0c6a9e0aad05c331acf}{Touch\+Control.\+touch\+Id}) and touch phases (\doxylink{class_unity_engine_1_1_input_system_1_1_controls_1_1_touch_control_ae10105e977a25050c71bdad2f78203c6}{Touch\+Control.\+phase}) in order to tell one touch apart from another.

Also, this class protects against losing touches. If a touch is shorter-\/lived than a single input update, \doxylink{class_unity_engine_1_1_input_system_1_1_touchscreen}{Touchscreen} may overwrite it with a new touch coming in in the same update whereas this class will retain all changes that happened on the touchscreen in any particular update.

The API makes a distinction between "{}fingers"{} and "{}touches"{}. A touch refers to one contact state change event, that is, a finger beginning to touch the screen (\doxylink{namespace_unity_engine_1_1_input_system_a586a6521911c62d23469e51a336d29dba7b81520329371177ede3ccbc307bc85d}{Touch\+Phase.\+Began}), moving on the screen (\doxylink{namespace_unity_engine_1_1_input_system_a586a6521911c62d23469e51a336d29dba95bfdefaaf01553cbf21ccd942855c69}{Touch\+Phase.\+Moved}), or being lifted off the screen (\doxylink{namespace_unity_engine_1_1_input_system_a586a6521911c62d23469e51a336d29dba0157fbb99be9622bdaa6e544270c0622}{Touch\+Phase.\+Ended} or \doxylink{namespace_unity_engine_1_1_input_system_a586a6521911c62d23469e51a336d29dba0e22fe7d45f8e5632a4abf369b24e29c}{Touch\+Phase.\+Canceled}). A finger, on the other hand, always refers to the Nth contact on the screen.

A \doxylink{struct_unity_engine_1_1_input_system_1_1_enhanced_touch_1_1_touch}{Touch} instance is a struct which only contains a reference to the actual data which is stored in unmanaged memory.

All touches that are either on-\/going as of the current frame or have ended in the current frame. 

A touch that begins in a frame will always have its phase set to \doxylink{namespace_unity_engine_1_1_input_system_a586a6521911c62d23469e51a336d29dba7b81520329371177ede3ccbc307bc85d}{Touch\+Phase.\+Began} even if there was also movement (or even an end/cancellation) for the touch in the same frame.

A touch that begins and ends in the same frame will have its \doxylink{namespace_unity_engine_1_1_input_system_a586a6521911c62d23469e51a336d29dba7b81520329371177ede3ccbc307bc85d}{Touch\+Phase.\+Began} surface in that frame and then another entry with \doxylink{namespace_unity_engine_1_1_input_system_a586a6521911c62d23469e51a336d29dba0157fbb99be9622bdaa6e544270c0622}{Touch\+Phase.\+Ended} surface in the {\itshape next} frame. This logic implies that there can be more active touches than concurrent touches supported by the hardware/platform.

A touch that begins and moves in the same frame will have its \doxylink{namespace_unity_engine_1_1_input_system_a586a6521911c62d23469e51a336d29dba7b81520329371177ede3ccbc307bc85d}{Touch\+Phase.\+Began} surface in that frame and then another entry with \doxylink{namespace_unity_engine_1_1_input_system_a586a6521911c62d23469e51a336d29dba95bfdefaaf01553cbf21ccd942855c69}{Touch\+Phase.\+Moved} and the screen motion surface in the {\itshape next} frame {\itshape except} if the touch also ended in the frame (in which case \doxylink{struct_unity_engine_1_1_input_system_1_1_enhanced_touch_1_1_touch_af4d520649da3c818adf566d9c5456193}{phase} will be \doxylink{namespace_unity_engine_1_1_input_system_a586a6521911c62d23469e51a336d29dba0157fbb99be9622bdaa6e544270c0622}{Touch\+Phase.\+Ended} instead of \doxylink{namespace_unity_engine_1_1_input_system_a586a6521911c62d23469e51a336d29dba95bfdefaaf01553cbf21ccd942855c69}{Touch\+Phase.\+Moved}).

Note that the touches reported by this API do {\itshape not} necessarily have to match the contents of Unity\+Engine.\+Input.\+touches

. The reason for this is that the {\ttfamily Unity\+Engine.\+Input} API and the Input \doxylink{namespace_system}{System} API flush their input queues at different points in time and may thus have a different view on available input. In particular, the Input \doxylink{namespace_system}{System} event queue is flushed {\itshape later} in the frame than inputs for {\ttfamily Unity\+Engine.\+Input} and may thus have newer inputs available. On Android, for example, touch input is gathered from a separate \doxylink{namespace_unity_engine_1_1_u_i}{UI} thread and fed into the input system via a "{}background"{} event queue that can gather input asynchronously. Due to this setup, touch events that will reach {\ttfamily Unity\+Engine.\+Input} only in the next frame may have already reached the Input \doxylink{namespace_system}{System}.

bool Unity\+Engine.\+Input\+System.\+Enhanced\+Touch.\+Touch.\+valid\hspace{0.3cm}{\ttfamily [get]}}



Whether this touch record holds valid data. 

If true, the data contained in the touch is valid.

\doxylink{struct_unity_engine_1_1_input_system_1_1_enhanced_touch_1_1_touch}{Touch} data is stored in unmanaged memory as a circular input buffer. This means that when the buffer runs out of capacity, older touch entries will get reused. When this happens, existing {\ttfamily \doxylink{struct_unity_engine_1_1_input_system_1_1_enhanced_touch_1_1_touch}{Touch}} instances referring to the record become invalid.

This property can be used to determine whether the record held on to by the {\ttfamily \doxylink{struct_unity_engine_1_1_input_system_1_1_enhanced_touch_1_1_touch}{Touch}} instance is still valid.

This property will be {\ttfamily false} for default-\/initialized {\ttfamily \doxylink{struct_unity_engine_1_1_input_system_1_1_enhanced_touch_1_1_touch}{Touch}} instances.

Note that accessing most of the other properties on this struct when the touch is invalid will trigger {\ttfamily Invalid\+Operation\+Exception}.

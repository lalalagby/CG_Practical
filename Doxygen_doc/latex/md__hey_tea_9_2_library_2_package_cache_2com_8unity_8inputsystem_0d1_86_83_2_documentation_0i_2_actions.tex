{\bfseries{Actions}} allow you to separate the logical meaning of an input (the things your user can do in your game or app, such as move, jump, crouch) and the device-\/specific controls (for example, pressing a button or moving a gamepad stick).

Without Actions, the meaning of your input and the device controls end up hard-\/coded together in your scripts, which although quick to implement, is not very flexible and therefore not always desirable. For example, here the move\+Vector variable is hard-\/coded to read values from the right stick of a gamepad\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{\textcolor{comment}{//\ without\ Actions,\ you\ end\ up\ hard-\/coding\ device\ controls}}
\DoxyCodeLine{\textcolor{comment}{//\ like\ this:\ (not\ always\ desirable)}}
\DoxyCodeLine{moveVector\ =\ gamepad.rightStick.ReadValue<Vector2>();}

\end{DoxyCode}


When you use Actions, you do not need to refer to specific devices or their controls in your code. Instead the Action\textquotesingle{}s binding defines which controls on which device is used to perform the action, and your code becomes simpler. In this example, {\ttfamily move\+Action} would contain a reference to an Action, defined either in code, or in an Action Asset\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{\textcolor{comment}{//\ using\ Actions\ removes\ the\ need\ to\ refer\ directly\ to\ device\ controls}}
\DoxyCodeLine{moveAmount\ =\ moveAction.ReadValue<Vector2>();}

\end{DoxyCode}


You can then use the visual editor (either in the inspector, or in the Actions Asset editor) to establish the mapping between the Action and one or more device controls. For example in this screenshot, the "{}move"{} action is bound to the left gamepad stick, and the keyboard\textquotesingle{}s arrow keys.



This also makes it easier to let players \href{ActionBindings.md\#interactive-rebinding}{\texttt{ customize bindings at runtime}}.

Actions use {\ttfamily Input\+Binding} to refer to the inputs they collect. For more information about Bindings and how to use them, see \doxysectlink{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_action_bindings}{Action Bindings}{0}.

Each Action has a name (\href{../api/UnityEngine.InputSystem.InputAction.html\#UnityEngine_InputSystem_InputAction_name}{\texttt{ {\ttfamily Input\+Action.\+name}}}), which must be unique within the Action Map that the Action belongs to, if any (see \href{../api/UnityEngine.InputSystem.InputAction.html\#UnityEngine_InputSystem_InputAction_actionMap}{\texttt{ {\ttfamily Input\+Action.\+action\+Map}}}). Each Action also has a unique ID (\href{../api/UnityEngine.InputSystem.InputAction.html\#UnityEngine_InputSystem_InputAction_id}{\texttt{ {\ttfamily Input\+Action.\+id}}}), which you can use to reference the Action. The ID remains the same even if you rename the Action.

Each Action Map has a name (\href{../api/UnityEngine.InputSystem.InputActionMap.html\#UnityEngine_InputSystem_InputActionMap_name}{\texttt{ {\ttfamily Input\+Action\+Map.\+name}}}), which must be unique within the Action Asset that the Action Map belongs to, if any (see \href{../api/UnityEngine.InputSystem.InputActionMap.html\#UnityEngine_InputSystem_InputActionMap_asset}{\texttt{ {\ttfamily Input\+Action\+Map.\+asset}}}). Each Action Map also has a unique ID (\href{../api/UnityEngine.InputSystem.InputActionMap.html\#UnityEngine_InputSystem_InputActionMap_id}{\texttt{ {\ttfamily Input\+Action\+Map.\+id}}}), which you can use to reference the Action Map. When you enable an Action, the Input \doxylink{namespace_system}{System} resolves its bindings, unless it has done so already, or if the set of devices that the Action can use has not changed. For more details about this process, see the documentation on \href{ActionBindings.md\#binding-resolution}{\texttt{ binding resolution}}.

You can\textquotesingle{}t change certain aspects of the configuration, such Action Bindings, while an Action is enabled. To stop Actions or Action Maps from responding to input, call \href{../api/UnityEngine.InputSystem.InputAction.html\#UnityEngine_InputSystem_InputAction_Disable}{\texttt{ {\ttfamily Disable}}}.

While enabled, an Action actively monitors the \doxysectlink{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_controls}{Control(s)}{0} it\textquotesingle{}s bound to. If a bound Control changes state, the Action processes the change. If the Control\textquotesingle{}s change represents an \doxysectlink{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_interactions}{Interaction}{0} change, the Action creates a response. All of this happens during the Input \doxylink{namespace_system}{System} update logic. Depending on the \href{Settings.md\#update-mode}{\texttt{ update mode}} selected in the input settings, this happens once every frame, once every fixed update, or manually if updates are set to manual.\hypertarget{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_actions_autotoc_md1275}{}\doxysubsection{\texorpdfstring{Responding to Actions}{Responding to Actions}}\label{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_actions_autotoc_md1275}
An Action doesn\textquotesingle{}t represent an actual response to input by itself. Instead, an Action informs your code that a certain type of input has occurred. Your code then responds to this information.

There are several ways to do this\+:


\begin{DoxyEnumerate}
\item Each Action has a \`{}started\`{}, \`{}performed\`{}, and \`{}canceled\`{} callback.
\item Each Action Map has an \`{}action\+Triggered\`{} callback.
\item The Input \doxylink{namespace_system}{System} has a global \`{}\+Input\+System.on\+Action\+Change\`{} callback.
\item You can poll the current state of an Action whenever you need it.
\item \`{}\+Input\+Action\+Trace\`{} can record changes happening on Actions.
\end{DoxyEnumerate}

There are also two higher-\/level, more streamlined ways of picking up input from Actions\+: use \href{PlayerInput.md\#notification-behaviors}{\texttt{ {\ttfamily Player\+Input}}}, or \href{ActionAssets.md\#auto-generating-script-code-for-actions}{\texttt{ generate script code}} that wraps around the Input Actions.\hypertarget{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_actions_autotoc_md1276}{}\doxysubsubsection{\texorpdfstring{Action callbacks}{Action callbacks}}\label{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_actions_autotoc_md1276}

Once recorded, a trace can be safely read from multiple threads as long as it is not concurrently being written to and as long as the Action setup (that is, the configuration data accessed by the trace) is not concurrently being changed on the main thread.\hypertarget{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_actions_autotoc_md1281}{}\doxysubsection{\texorpdfstring{Action types}{Action types}}\label{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_actions_autotoc_md1281}
Each Action can be one of three different \href{../api/UnityEngine.InputSystem.InputActionType.html}{\texttt{ Action types}}. You can select the Action type in the Input Action editor window, or by specifying the {\ttfamily type} parameter when calling the \href{../api/UnityEngine.InputSystem.InputAction.html\#UnityEngine_InputSystem_InputAction__ctor_System_String_UnityEngine_InputSystem_InputActionType_System_String_System_String_System_String_System_String_}{\texttt{ {\ttfamily Input\+Action()}}} constructor. The Action type influences how the Input \doxylink{namespace_system}{System} processes state changes for the Action. The default Action type is {\ttfamily Value}.\hypertarget{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_actions_autotoc_md1282}{}\doxysubsubsection{\texorpdfstring{Value}{Value}}\label{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_actions_autotoc_md1282}
This is the default Action type. Use this for any inputs which should track continuous changes to the state of a Control.

\href{../api/UnityEngine.InputSystem.InputActionType.html\#UnityEngine_InputSystem_InputActionType_Value}{\texttt{ {\ttfamily Value}}} type actions continuously monitor all the Controls which are bound to the Action, and then choose the one which is the most actuated to be the Control driving the Action, and report the values from that Control in callbacks, triggered whenever the value changes. If a different bound Control actuated more, then that Control becomes the Control driving the Action, and the Action starts reporting values from that Control. This process is called \href{ActionBindings.md\#conflicting-inputs}{\texttt{ conflict resolution}}. This is useful if you want to allow different Controls to control an Action in the game, but only take input from one Control at the same time.

When the Action initially enables, it performs an \href{ActionBindings.md\#initial-state-check}{\texttt{ initial state check}} of all bound Controls. If any of them is actuated, the Action then triggers a callback with the current value.\hypertarget{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_actions_autotoc_md1283}{}\doxysubsubsection{\texorpdfstring{Button}{Button}}\label{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_actions_autotoc_md1283}
This is very similar to \href{../api/UnityEngine.InputSystem.InputActionType.html\#UnityEngine_InputSystem_InputActionType_Value}{\texttt{ {\ttfamily Value}}}, but \href{../api/UnityEngine.InputSystem.InputActionType.html\#UnityEngine_InputSystem_InputActionType_Button}{\texttt{ {\ttfamily Button}}} type Actions can only be bound to \href{../api/UnityEngine.InputSystem.Controls.ButtonControl.html}{\texttt{ {\ttfamily Button\+Control}}} Controls, and don\textquotesingle{}t perform an initial state check like \href{../api/UnityEngine.InputSystem.InputActionType.html\#UnityEngine_InputSystem_InputActionType_Value}{\texttt{ {\ttfamily Value}}} Actions do (see the Value section above). Use this for inputs that trigger an Action once every time they are pressed. The initial state check is usually not useful in such cases, because it can trigger actions if the button is still held down from a previous press when the Action was enabled.\hypertarget{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_actions_autotoc_md1284}{}\doxysubsubsection{\texorpdfstring{Pass-\/\+Through}{Pass-\/\+Through}}\label{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_actions_autotoc_md1284}
\href{../api/UnityEngine.InputSystem.InputActionType.html\#UnityEngine_InputSystem_InputActionType_PassThrough}{\texttt{ {\ttfamily Pass-\/\+Through}}} Actions bypass the \href{ActionBindings.md\#conflicting-inputs}{\texttt{ conflict resolution}} process described above for {\ttfamily Value} Actions and don\textquotesingle{}t use the concept of a specific Control driving the Action. Instead, any change to any bound Control triggers a callback with that Control\textquotesingle{}s value. This is useful if you want to process all input from a set of Controls.\hypertarget{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_actions_autotoc_md1285}{}\doxysubsection{\texorpdfstring{Debugging Actions}{Debugging Actions}}\label{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_actions_autotoc_md1285}
To see currently enabled Actions and their bound Controls, use the \href{Debugging.md\#debugging-actions}{\texttt{ Input Debugger}}.

You can also use the \href{Debugging.md\#inputactionvisualizer}{\texttt{ {\ttfamily Input\+Action\+Visualizer}}} component from the Visualizers sample to get an on-\/screen visualization of an Action\textquotesingle{}s value and Interaction state in real-\/time.\hypertarget{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_actions_autotoc_md1286}{}\doxysubsection{\texorpdfstring{Using Actions with multiple players}{Using Actions with multiple players}}\label{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_actions_autotoc_md1286}

\doxysubsection{Detailed Description}
A control layout specifies the composition of an \doxylink{class_unity_engine_1_1_input_system_1_1_input_control}{Input\+Control} or \doxylink{class_unity_engine_1_1_input_system_1_1_input_device}{Input\+Device}. 

Control layouts can be created in three possible ways\+:


\begin{DoxyEnumerate}
\item Loaded from JSON. 
\item Constructed through reflection from \doxylink{}{Input\+Controls} classes. 
\item Through layout factories using \doxylink{class_unity_engine_1_1_input_system_1_1_layouts_1_1_input_control_layout_1_1_builder}{Input\+Control\+Layout.\+Builder}. 
\end{DoxyEnumerate}

Once constructed, control layouts are immutable (but you can always replace a registered layout in the system and it will affect everything constructed from the layout).

Control layouts can be for arbitrary control rigs or for entire devices. Device layouts can be matched to \doxylink{struct_unity_engine_1_1_input_system_1_1_layouts_1_1_input_device_description}{device description} using associated \doxylink{struct_unity_engine_1_1_input_system_1_1_layouts_1_1_input_device_matcher}{device matchers}.

\doxylink{class_unity_engine_1_1_input_system_1_1_layouts_1_1_input_control_layout}{Input\+Control\+Layout} objects are considered temporaries. Except in the editor, they are not kept around beyond device creation.

See the \href{../manual/Layouts.html}{\texttt{ manual}} for more details on control layouts.

Return the type of values produced by controls created from the layout. 

\begin{DoxyReturn}{Returns}
The value type of the control or null if it cannot be determined.
\end{DoxyReturn}


This method only returns the statically inferred value type. This type corresponds to the type argument to Input\+Control$<$\+TValue$>$ in the inheritance hierarchy of type. As the type used by the layout may not inherit from Input\+Control$<$\+TValue$>$, this may mean that the value type cannot be inferred and the method will return null.

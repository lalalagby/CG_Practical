\doxysubsection{Detailed Description}
Metadata for an input device. 

Device descriptions are mainly used to determine which \doxylink{class_unity_engine_1_1_input_system_1_1_layouts_1_1_input_control_layout}{Input\+Control\+Layout} to create an actual \doxylink{class_unity_engine_1_1_input_system_1_1_input_device}{Input\+Device} instance from. Each description is comprised of a set of properties that each are individually optional. However, for a description to be usable, at least some need to be set. Generally, the minimum viable description for a device is one with \doxylink{struct_unity_engine_1_1_input_system_1_1_layouts_1_1_input_device_description_a3f520f70f0a42b21e62ee89576afb692}{device\+Class} filled out.


\begin{DoxyCode}{0}
\DoxyCodeLine{\textcolor{comment}{//\ Device\ description\ equivalent\ to\ a\ generic\ gamepad\ with\ no}}
\DoxyCodeLine{\textcolor{comment}{//\ further\ information\ about\ the\ device.}}
\DoxyCodeLine{\textcolor{keyword}{new}\ InputDeviceDescription}
\DoxyCodeLine{\{}
\DoxyCodeLine{\ \ \ \ \mbox{\hyperlink{struct_unity_engine_1_1_input_system_1_1_layouts_1_1_input_device_description_a3f520f70f0a42b21e62ee89576afb692}{deviceClass}}\ =\ \textcolor{stringliteral}{"{}Gamepad"{}}}
\DoxyCodeLine{\};}

\end{DoxyCode}
 

Device descriptions will usually be supplied by the \doxylink{namespace_unity}{Unity} runtime but can also be manually fed into the system using Input\+System.\+Add\+Device(\+Input\+Device\+Description). The system will remember each device description it has seen regardless of whether it was able to successfully create a device from the description. To query the list of descriptions that for whatever reason did not result in a device being created, call Input\+System.\+Get\+Unsupported\+Devices().

Whenever layout registrations in the system are changed (e.\+g. by calling Input\+System.\+Register\+Layout$<$\+T$>$ or whenever \doxylink{class_unity_engine_1_1_input_system_1_1_input_settings_aa104b89e505dd3c8d30feab6b27c1b42}{Input\+Settings.\+supported\+Devices} is changed, the system will go through the list of unsupported devices itself and figure out if there are device descriptions that now it can turn into devices. The same also applies in reverse; if, for example, a layout is removed that is currently used a device, the device will be removed and its description (if any) will be placed on the list of unsupported devices.

string Unity\+Engine.\+Input\+System.\+Layouts.\+Input\+Device\+Description.\+capabilities\hspace{0.3cm}{\ttfamily [get]}, {\ttfamily [set]}}



An optional JSON string listing device-\/specific capabilities. 

Interface-\/specific listing of device capabilities.

The primary use of this field is to allow custom layout factories to create layouts on the fly from in-\/depth device descriptions delivered by external APIs.

In the case of \doxylink{namespace_unity_engine_1_1_input_system_1_1_h_i_d}{HID}, for example, this field contains a JSON representation of the \doxylink{namespace_unity_engine_1_1_input_system_1_1_h_i_d}{HID} descriptor (see \doxylink{struct_unity_engine_1_1_input_system_1_1_h_i_d_1_1_h_i_d_1_1_h_i_d_device_descriptor}{HID.\+HID.\+HIDDevice\+Descriptor}) as reported by the device driver. This descriptor contains information about all I/O elements on the device which can be used to determine the control setup and data format used by the device.

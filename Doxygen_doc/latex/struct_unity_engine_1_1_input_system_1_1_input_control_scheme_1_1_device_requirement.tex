\doxysubsection{Detailed Description}
Note that device requirements may require specific controls to be present rather than only requiring the presence of a certain type of device. For example, a requirement with a \doxylink{struct_unity_engine_1_1_input_system_1_1_input_control_scheme_1_1_device_requirement_ac81f846c8c88370483b3a8137d97b1d6}{control\+Path} of "{}\texorpdfstring{$\ast$}{*}/\{\+Primary\+Action\}"{} will be satisfied by any device that has a control marked as Common\+Usages.\+Primary\+Action.

Requirements are ordered in a list and can combine with their previous requirement in either \doxylink{struct_unity_engine_1_1_input_system_1_1_input_control_scheme_1_1_device_requirement_a9c6aa378e873c3e5e55a9a3fdbc14e89}{AND} or in \doxylink{struct_unity_engine_1_1_input_system_1_1_input_control_scheme_1_1_device_requirement_a45f76a7e7fe614e8bb87eeeaec03f76b}{OR} fashion. The default is for requirements to combine with AND.

Note that it is not possible to express nested constraints like {\ttfamily (a AND b) OR (c AND d)}. Also note that operator precedence is the opposite of C\#, meaning that OR has {\itshape higher} precedence than AND. This means that {\ttfamily a OR b AND c OR d} reads as {\ttfamily (a OR b) AND (c OR d)} (in C\# it would read as {\ttfamily a OR (b AND c) OR d}.

More complex expressions can often be expressed differently. For example, {\ttfamily (a AND b) OR (c AND d)} can be expressed as {\ttfamily a OR c AND b OR d}.

bool Unity\+Engine.\+Input\+System.\+Input\+Control\+Scheme.\+Device\+Requirement.\+is\+OR\hspace{0.3cm}{\ttfamily [get]}, {\ttfamily [set]}}



Whether the requirement combines with the previous requirement (if any) as a boolean OR. 

This allows designing control schemes that flexibly work with combinations of devices such that if one specific device isn\textquotesingle{}t present, another device can substitute for it.

For example, to design a mouse+keyboard control scheme that can alternatively work with a pen instead of a mouse, the first requirement could be for "{}<\+Keyboard>"{}, the second one could be for "{}<\+Mouse>"{} and the third one could be for "{}<\+Pen>"{} and return true for this property. Both the mouse and the pen would be marked as required (i.\+e. not \doxylink{struct_unity_engine_1_1_input_system_1_1_input_control_scheme_1_1_device_requirement_a2b1ca7b1009e8f63b7ac2191075e815b}{is\+Optional}) but the device requirements are satisfied even if either device is present.

Note that if both a pen and a mouse are present at the same time, still only one device is picked. In this case, the mouse "{}wins"{} as it comes first in the list of requirements.

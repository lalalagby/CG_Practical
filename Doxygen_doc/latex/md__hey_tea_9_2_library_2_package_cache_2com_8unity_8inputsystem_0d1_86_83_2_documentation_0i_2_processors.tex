 An Input Processor takes a value and returns a processed result for it. The received value and result value must be of the same type. For example, you can use a clamp Processor to clamp values from a control to a certain range.

\texorpdfstring{$>$}{>}{\bfseries{Note}}\+: To convert received input values into different types, see \href{ActionBindings.md\#composite-bindings}{\texttt{ composite Bindings}}.

You can install Processors on \doxysectlink{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_action_bindings}{bindings}{0}, \doxysectlink{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_actions}{actions}{0} or on \doxysectlink{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_controls}{controls}{0}.

Each Processor is \href{../api/UnityEngine.InputSystem.InputSystem.html\#UnityEngine_InputSystem_InputSystem_RegisterProcessor__1_System_String_}{\texttt{ registered}} using a unique name. To replace an existing Processor, register your own Processor under an existing name.

Processors can have parameters which can be booleans, integers, or floating-\/point numbers. When created in data such as \doxysectlink{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_action_bindings}{bindings}{0}, processors are described as strings that look like function calls\+:

\hypertarget{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_processors_autotoc_md1527}{}\doxysubsection{\texorpdfstring{Processors on Bindings}{Processors on Bindings}}\label{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_processors_autotoc_md1527}
When you create Bindings for your \doxysectlink{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_actions}{actions}{0}, you can choose to add Processors to the Bindings. These process the values from the controls they bind to, before the system applies them to the Action value. For instance, you might want to invert the {\ttfamily Vector2} values from the controls along the Y axis before passing these values to the Action that drives the input logic for your application. To do this, you can add an Invert Vector2 Processor to your Binding.

If you\textquotesingle{}re using \doxysectlink{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_action_assets}{Input Action Assets}{0}, you can add any Processor to your Bindings in the Input Action editor. Select the Binding you want to add Processors to so that the right pane of the window displays the properties for that Binding. Select the Add (+) icon on the {\bfseries{Processors}} foldout to open a list of all available Processors that match your control type, then choose a Processor type to add a Processor instance of that type. The Processor now appears under the {\bfseries{Processors}} foldout. If the Processor has any parameters, you can edit them in the {\bfseries{Processors}} foldout.



To remove a Processor, click the Remove (-\/) icon next to it. You can also use the up and down arrows to change the order of Processors. This affects the order in which the system processes values.

If you create your Bindings in code, you can add Processors like this\+:

\DoxyCodeLine{\ \ \ \ .WithProcessor(\textcolor{stringliteral}{"{}invertVector2(invertX=false)"{}});}

\end{DoxyCode}

Processors on Actions work in the same way as Processors on Bindings, but they affect all controls bound to an Action, rather than just the controls from a specific Binding. If there are Processors on both the Binding and the Action, the system processes the ones from the Binding first.

You can add and edit Processors on Actions in \doxysectlink{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_action_assets}{Input Action Assets}{0} the same way as you would for Bindings\+: select an Action to edit, then add one or more Processors in the right window pane.

If you create your Actions in code, you can add Processors like this\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{var\ action\ =\ \textcolor{keyword}{new}\ InputAction(processors:\ \textcolor{stringliteral}{"{}invertVector2(invertX=false)"{}});}

\end{DoxyCode}
\hypertarget{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_processors_autotoc_md1529}{}\doxysubsection{\texorpdfstring{Processors on Controls}{Processors on Controls}}\label{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_processors_autotoc_md1529}
You can have any number of Processors directly on an \href{../api/UnityEngine.InputSystem.InputControl.html}{\texttt{ {\ttfamily Input\+Control}}}, which then process the values read from the Control. Whenever you call \href{../api/UnityEngine.InputSystem.InputControl-1.html\#UnityEngine_InputSystem_InputControl_1_ReadValue}{\texttt{ {\ttfamily Read\+Value}}} on a Control, all Processors on that Control process the value before it gets returned to you. You can use \href{../api/UnityEngine.InputSystem.InputControl-1.html\#UnityEngine_InputSystem_InputControl_1_ReadUnprocessedValue}{\texttt{ {\ttfamily Read\+Unprocessed\+Value}}} on a Control to bypass the Processors.

The Input \doxylink{namespace_system}{System} adds Processors to a Control during device creation, if they\textquotesingle{}re specified in the Control\textquotesingle{}s \doxysectlink{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_layouts}{layout}{0}. You can\textquotesingle{}t add Processors to existing Controls after they\textquotesingle{}ve been created, so you can only add Processors to Controls when you\textquotesingle{}re \href{Devices.md\#creating-custom-devices}{\texttt{ creating custom devices}}. The devices that the Input \doxylink{namespace_system}{System} supports out of the box already have some useful Processors added on their Controls. For instance, sticks on gamepads have a Stick Deadzone Processor.

If you\textquotesingle{}re using a layout generated by the Input \doxylink{namespace_system}{System} from a \href{Devices.md\#step-1-the-state-struct}{\texttt{ state struct}} using \href{../api/UnityEngine.InputSystem.Layouts.InputControlAttribute.html}{\texttt{ {\ttfamily Input\+Control\+Attributes}}}, you can specify the Processors you want to use via the \href{../api/UnityEngine.InputSystem.Layouts.InputControlAttribute.html\#UnityEngine_InputSystem_Layouts_InputControlAttribute_processors}{\texttt{ {\ttfamily processors}}} property of the attribute, like this\+:

You can also write custom Processors to use in your Project. Custom Processors are available in the UI and code in the same way as the built-\/in Processors. Add a class derived from \href{../api/UnityEngine.InputSystem.InputProcessor-1.html}{\texttt{ {\ttfamily Input\+Processor\texorpdfstring{$<$}{<}TValue\texorpdfstring{$>$}{>}}}}, and implement the \href{../api/UnityEngine.InputSystem.InputProcessor-1.html\#UnityEngine_InputSystem_InputProcessor_1_Process__0_UnityEngine_InputSystem_InputControl_}{\texttt{ {\ttfamily Process}}} method\+:

\texorpdfstring{$>$}{>}{\bfseries{IMPORTANT}}\+: Processors must be {\bfseries{stateless}}. This means you cannot store local state in a processor that will change depending on the input being processed. The reason for this is because processors are not part of the \href{./Controls.md\#control-state}{\texttt{ input state}} that the Input \doxylink{namespace_system}{System} keeps.

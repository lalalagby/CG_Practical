To disable \doxylink{namespace_burst}{Burst}\textquotesingle{}s safety check code, use \href{xref:Unity.Burst.BurstCompileAttribute.DisableSafetyChecks}{\texttt{ Disable\+Safety\+Checks}}. This results in faster code generation, however make sure that you use containers in a safe fashion.

To disable safety checks on a job or function-\/pointer set {\ttfamily Disable\+Safety\+Checks} to {\ttfamily true}\+:

\doxylink{namespace_burst}{Burst} ignores code marked explicitly with {\ttfamily Disable\+Safety\+Checks = true} when it safety checks your code if you set {\bfseries{Enable Safety Checks}} to {\bfseries{On}} in the Editor. Select {\bfseries{Force On}} to make \doxylink{namespace_burst}{Burst} to safety check all code, including code marked with {\ttfamily Disable\+Safety\+Checks = true}.

When you enable the {\bfseries{Show Timings}} setting, \doxylink{namespace_unity}{Unity} logs an output in the \href{https://docs.unity3d.com/Manual/Console.html}{\texttt{ Console window}} for each library of entry points that \doxylink{namespace_burst}{Burst} compiles. \doxylink{namespace_burst}{Burst} batches the compilation into units of methods-\/per-\/assembly, and groups multiple entry-\/points together in a single compilation task. This output is useful if you want to report outliers in compilation to the \doxylink{namespace_burst}{Burst} compiler team (via the \href{https://forum.unity.com/forums/burst.629/}{\texttt{ Burst forum}}).

\doxylink{namespace_unity}{Unity} splits \doxylink{namespace_burst}{Burst}\textquotesingle{}s output into the following major sections\+:


\begin{DoxyItemize}
\item Method discovery (where \doxylink{namespace_burst}{Burst} works out what it needs to compile)
\item Front end (where \doxylink{namespace_burst}{Burst} turns C\# IL into an LLVM IR module)
\item Middle end (where \doxylink{namespace_burst}{Burst} specializes, optimizes, and cleans up the module)
\item Back-\/end (where \doxylink{namespace_burst}{Burst} turns the LLVM IR module into a native DLL)
\end{DoxyItemize}

The compile time in the front end and optimizer is linear to the amount operations that it needs to compile. More functions and more instructions means a longer compile time. The more generic functions you have, the higher the front end performance timings, because generic resolutions have non-\/zero costs.

The compile time in the back-\/end scales with the number of entry-\/points in the module. This is because each entry point is in its own native object file.

If the optimizer takes a significant amount of time, use {\ttfamily \mbox{[}Burst\+Compile(Optimize\+For = Optimize\+For.\+Fast\+Compilation)\mbox{]}} which reduces the optimizations that \doxylink{namespace_burst}{Burst} does, but compiles things much faster. Profile the job before and after to make sure that this tradeoff is right for that entry-\/point.

\doxysubsection{Detailed Description}
A typed and named source of input values in a hierarchy of controls. 

Base class for input controls with a specific value type.

\doxylink{namespace_unity_engine_1_1_input_system_1_1_controls}{Controls} can have children which in turn may have children. At the root of the child hierarchy is always an \doxylink{class_unity_engine_1_1_input_system_1_1_input_device}{Input\+Device} (which themselves are Input\+Controls).

\doxylink{namespace_unity_engine_1_1_input_system_1_1_controls}{Controls} can be looked up by their \doxylink{class_unity_engine_1_1_input_system_1_1_input_control_a60d8e17e65b94f2cf7a0ce7f10af0075}{path} (see Input\+Control\+Path.\+Try\+Find\+Control).

Each control must have a unique \doxylink{class_unity_engine_1_1_input_system_1_1_input_control_a94f646dff1e9cbbbf84857bffa490274}{name} within the \doxylink{class_unity_engine_1_1_input_system_1_1_input_control_a7861c4cd07f6b756c6f22c18f195a3c1}{children} of its \doxylink{class_unity_engine_1_1_input_system_1_1_input_control_abce81da2e91c3b46a949185b66fff32d}{parent}. Multiple names can be assigned to controls using aliases (see aliases). Name lookup is case-\/insensitive.

For display purposes, a control may have a separate \doxylink{class_unity_engine_1_1_input_system_1_1_input_control_ac31c2a2f570e527aacc86ee8060e2ecc}{display\+Name}. This name will usually correspond to what the control is caused on the actual underlying hardware. For example, on an Xbox gamepad, the control with the name "{}button\+South"{} will have a display name of "{}\+A"{}. \doxylink{namespace_unity_engine_1_1_input_system_1_1_controls}{Controls} that have very long display names may also have a \doxylink{class_unity_engine_1_1_input_system_1_1_input_control_ab4716691d15fffa36ab61fec8c05c6ac}{short\+Display\+Name}. This is the case for the "{}\+Left Button"{} on the \doxylink{class_unity_engine_1_1_input_system_1_1_mouse}{Mouse}, for example, which is commonly abbreviated "{}\+LMB"{}.

In addition to names, a control may have usages associated with it (see \doxylink{class_unity_engine_1_1_input_system_1_1_input_control_a09e4b347afac3558d9dbe4c4919457b8}{usages}). A usage indicates how a control is meant to be used. For example, a button can be assigned the "{}\+Primary\+Action"{} usage to indicate it is the primary action button the device. Within a device, usages have to be unique. See Common\+Usages for a list of standardized usages.

\doxylink{namespace_unity_engine_1_1_input_system_1_1_controls}{Controls} do not actually store values. Instead, every control receives an Input\+State\+Block which, after the control\textquotesingle{}s device has been added to the system, is used to read out values from the device\textquotesingle{}s backing store. This backing store is referred to as "{}state"{} in the API as opposed to "{}values"{} which represent the data resulting from reading state. The format that each control stores state in is specific to the control. It can vary not only between controls of different types but also between controls of the same type. An Axis\+Control, for example, can be stored as a float or as a byte or in a number of other formats. state\+Block identifies both where the control stores its state as well as the format it stores it in.

\doxylink{namespace_unity_engine_1_1_input_system_1_1_controls}{Controls} are generally not created directly but are created internally by the input system from data known as "{}layouts"{} (see Input\+Control\+Layout). Each such layout describes the setup of a specific hierarchy of controls. The system internally maintains a registry of layouts and produces devices and controls from them as needed. The layout that a control has been created from can be queried using \doxylink{class_unity_engine_1_1_input_system_1_1_input_control_a8d349861ff83c7a74573bea9a2cf2886}{layout}. For most purposes, the intricacies of the control layout mechanisms can be ignored and it is sufficient to know the names of a small set of common device layouts such as "{}\+Keyboard"{}, "{}\+Mouse"{}, "{}\+Gamepad"{}, and "{}\+Touchscreen"{}.

Each control has a single, fixed value type. The type can be queried at runtime using \doxylink{class_unity_engine_1_1_input_system_1_1_input_control_aa863a140183822dd850659336c9667ba}{value\+Type}. Most types of controls are derived from Input\+Control$<$\+TValue$>$ which has APIs specific to the type of value of the control (e.\+g. \doxylink{class_unity_engine_1_1_input_system_1_1_input_control_a2130e1c517844e5ae729eba1cd798c94}{Input\+Control$<$\+TValue$>$.\+Read\+Value()}.

This can only be called on devices that have been added to the system (\doxylink{class_unity_engine_1_1_input_system_1_1_input_device_a4ff5e7586f91a321108317341566d4eb}{Input\+Device.\+added}).

If internal feature "{}\+USE\+\_\+\+READ\+\_\+\+VALUE\+\_\+\+CACHING"{} is enabled, then this property implements caching to avoid applying processors when the underlying control has not changed. With this in mind, be aware of processors that use global state, such as the \doxylink{class_unity_engine_1_1_input_system_1_1_processors_1_1_axis_deadzone_processor}{Processors.\+Axis\+Deadzone\+Processor}. Unless the control unprocessed value has been changed, input system settings changed or \doxylink{class_unity_engine_1_1_input_system_1_1_input_control_ab30a6abcfbefca2dfe3666298f3df349}{Input\+Control.\+Apply\+Parameter\+Changes()} invoked, the processors will not run and calls to \doxylink{class_unity_engine_1_1_input_system_1_1_input_control_a190888e1bbbd636276392fb3b8625172}{value} will return the same result as previous calls.

If a processor requires to be run on every read, override \doxylink{class_unity_engine_1_1_input_system_1_1_input_processor_a7ebb134435afbd127a8cdc6b0a58ab89}{Input\+Processor.\+caching\+Policy} property in the processor and set it to \doxylink{class_unity_engine_1_1_input_system_1_1_input_processor_a9b5e79d60f62a8c8a5040d96fe207fa4a4be46f7df52f374eba4b58a7cddf070c}{Input\+Processor.\+Caching\+Policy.\+Evaluate\+On\+Every\+Read}.

To improve debugging try setting "{}\+PARANOID\+\_\+\+READ\+\_\+\+VALUE\+\_\+\+CACHING\+\_\+\+CHECKS"{} internal feature flag to check if cache value is still consistent. .

{\footnotesize\ttfamily \label{class_unity_engine_1_1_input_system_1_1_input_control_afa5a8531d7c9a030e8e484560a59fe6f} 
bool \mbox{\hyperlink{class_unity_engine_1_1_input_system_1_1_input_control}{Unity\+Engine.\+Input\+System.\+Input\+Control}}$<$ TValue $>$.noisy\hspace{0.3cm}{\ttfamily [get]}, {\ttfamily [set]}}



Whether the control is considered noisy. 

True if the control produces noisy input.

A control is considered "{}noisy"{} if it produces different values without necessarily requiring user interaction. A good example are sensors (see \doxylink{class_unity_engine_1_1_input_system_1_1_sensor}{Sensor}). For example, the PS4 controller which has a gyroscope sensor built into the device. Whereas sticks and buttons on the device require user interaction to produce non-\/default values, the gyro will produce varying values even if the device just sits there without user interaction.

The value of this property is determined by the layout (Input\+Control\+Layout) that the control has been built from.

Note that for devices (\doxylink{class_unity_engine_1_1_input_system_1_1_input_device}{Input\+Device}) this property is true if any control on the device is marked as noisy.

The primary effect of being noise is on \doxylink{class_unity_engine_1_1_input_system_1_1_input_device_a0a98203c32eb180ab8b7c862dc27635c}{Input\+Device.\+Make\+Current} and on interactive rebinding (see \doxylink{class_unity_engine_1_1_input_system_1_1_input_action_rebinding_extensions_1_1_rebinding_operation}{Input\+Action\+Rebinding\+Extensions.\+Rebinding\+Operation}). However, being noisy also affects automatic resetting of controls that happens when the application loses focus. While other controls are reset to their default value (except if {\ttfamily Application.\+run\+In\+Background} is true and the device the control belongs to is marked as \doxylink{class_unity_engine_1_1_input_system_1_1_input_device_acecc008067d46ee9a1db3cf059f6c709}{Input\+Device.\+can\+Run\+In\+Background}), noisy controls will not be reset but rather remain at their current value. This is based on the assumption that noisy controls most often represent sensor values and snapping the last sampling value back to default will usually have undesirable effects on an application\textquotesingle{}s simulation logic.

List of usage tags associated with the control. 

Usages apply "{}semantics"{} to a control. Whereas the name of a control identifies a particular "{}endpoint"{} within the control hierarchy, the usages of a control identify particular roles of specific control. A simple example is Common\+Usages.\+Back which identifies a control generally used to move backwards in the navigation history of a \doxylink{namespace_unity_engine_1_1_u_i}{UI}. On a keyboard, it is the escape key that generally fulfills this role whereas on a gamepad, it is generally the "{}\+B"{} / "{}\+Circle"{} button. Some devices may not have a control that generally fulfills this function and thus may not have any control with the "{}\+Back"{} usage.

By looking up controls by usage rather than by name, it is possible to locate the correct control to use for certain standardized situation without having to know the particulars of the device or platform.

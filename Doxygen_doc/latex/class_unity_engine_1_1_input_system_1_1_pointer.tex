override void \mbox{\hyperlink{class_unity_engine_1_1_input_system_1_1_pointer_af520df194ce0742bed345af0faf1b3f3}{Make\+Current}} ()
\begin{DoxyCompactList}\small\item\em Make this the current device of its type.

This method is called automatically by the input system when a device is added or when input is received on it. Many types of devices have {\ttfamily .current} getters that allow querying the last used device of a specific type directly (for example, see \doxylink{class_unity_engine_1_1_input_system_1_1_gamepad_a6f0bd4b71340da07b56f6ec56097da1d}{Gamepad.\+current}).

There is one special case, however, related to noise. A device that has noisy controls (i.\+e. controls for which \doxylink{class_unity_engine_1_1_input_system_1_1_input_control_afa5a8531d7c9a030e8e484560a59fe6f}{Input\+Control.\+noisy} is true) may receive input events that contain no meaningful user interaction but are simply just noise from the device. A good example of this is the PS4 gamepad which has a built-\/in gyro and may thus constantly feed events into the input system even if not being actually in use. If, for example, an Xbox gamepad and PS4 gamepad are both connected to a PC and the user is playing with the Xbox gamepad, the PS4 gamepad would still constantly make itself \doxylink{class_unity_engine_1_1_input_system_1_1_gamepad_a6f0bd4b71340da07b56f6ec56097da1d}{Gamepad.\+current} by simply flooding the system with events. Hence why by default, noise on {\ttfamily .current} getters will be filtered out and a device will only see {\ttfamily Make\+Current} getting called if there input was detected on non-\/noisy controls.  \end{DoxyCompactList}\end{DoxyCompactItemize}

\doxysubsection{Detailed Description}
Base class for pointer-\/style devices moving on a 2D screen. 

This class abstracts over general "{}pointing"{} behavior where a pointer is moved across a 2D surface. Operating at the {\ttfamily \doxylink{class_unity_engine_1_1_input_system_1_1_pointer}{Pointer}} level allows treating \doxylink{class_unity_engine_1_1_input_system_1_1_mouse}{Mouse}

, \doxylink{class_unity_engine_1_1_input_system_1_1_pen}{Pen}

, and \doxylink{class_unity_engine_1_1_input_system_1_1_touchscreen}{Touchscreen}

all as pointers with a set of shared behaviors.

Note that a pointer may have "{}multi-\/point"{} ability as is the case with multi-\/touch where multiple touches represent multiple concurrent "{}pointers"{}. However, for any pointer device with multiple pointers, only one pointer is considered "{}primary"{} and drives the pointer controls present on the base class.

Make this the current device of its type.

This method is called automatically by the input system when a device is added or when input is received on it. Many types of devices have {\ttfamily .current} getters that allow querying the last used device of a specific type directly (for example, see \doxylink{class_unity_engine_1_1_input_system_1_1_gamepad_a6f0bd4b71340da07b56f6ec56097da1d}{Gamepad.\+current}).

There is one special case, however, related to noise. A device that has noisy controls (i.\+e. controls for which \doxylink{class_unity_engine_1_1_input_system_1_1_input_control_afa5a8531d7c9a030e8e484560a59fe6f}{Input\+Control.\+noisy} is true) may receive input events that contain no meaningful user interaction but are simply just noise from the device. A good example of this is the PS4 gamepad which has a built-\/in gyro and may thus constantly feed events into the input system even if not being actually in use. If, for example, an Xbox gamepad and PS4 gamepad are both connected to a PC and the user is playing with the Xbox gamepad, the PS4 gamepad would still constantly make itself \doxylink{class_unity_engine_1_1_input_system_1_1_gamepad_a6f0bd4b71340da07b56f6ec56097da1d}{Gamepad.\+current} by simply flooding the system with events. Hence why by default, noise on {\ttfamily .current} getters will be filtered out and a device will only see {\ttfamily Make\+Current} getting called if there input was detected on non-\/noisy controls.

Note that some pointers have the ability to generate motion deltas {\itshape without} actually changing the position of the pointer. This is the case for \doxylink{class_unity_engine_1_1_input_system_1_1_mouse}{Mouse} which even when, for example, bumping up against the edges of the screen or when being locked in place, can generate motion. This means that activity on {\ttfamily delta} is not necessarily correlated with activity on \doxylink{class_unity_engine_1_1_input_system_1_1_pointer_a56648a46fa117cfe39f1bd3530c49700}{position}.

Deltas have two special behaviors attached to them that makes them quite unique among input controls.

For one, deltas will automatically reset to {\ttfamily (0,0)} between frames. If, for example, the current delta value is {\ttfamily (12,8)}, then after the next Input\+System.\+Update, the delta is automatically set to {\ttfamily (0,0)}. More precisely, deltas will reset as part of Input\+System.\+on\+Before\+Update. This happens every time regardless of whether there are pending motion events for the pointer or not. But because it happens in Input\+System.\+on\+Before\+Update (that is, {\itshape before} events are processed), subsequent motion deltas are incorporated normally.

Note that the resetting is visible to \doxylink{class_unity_engine_1_1_input_system_1_1_input_action}{Input\+Action}s. This means that when binding to a delta control from an action that is not using \doxylink{namespace_unity_engine_1_1_input_system_a0d0d044aae0aef15123e2f31f11ea94bae1ff6a7360e99cb46db6bb5d3ff8e2eb}{Input\+Action\+Type.\+Pass\+Through}, you will see the action getting cancelled at the start of every frame. With a {\ttfamily Pass\+Through} actions, you will instead see it perform one extra time with a zero value.

The other special behavior of deltas is accumulation. When receiving more than one motion update in a frame, deltas will not simply switch from one value to the other but instead accumulate them. For example, if two events are received for a pointer in a frame and one has a motion delta of {\ttfamily (1,1)} and the other has a motion delta of {\ttfamily (2,2)}, then once Input\+System.\+Update has finished processing events, the value of the delta control will be {\ttfamily (3,3)} and not {\ttfamily (2,2)}.

Note that just like resetting, accumulation is also visible to \doxylink{class_unity_engine_1_1_input_system_1_1_input_action}{Input\+Action}s. This means that because the delta control changes value twice, the action will trigger twice but the value when it is triggered the second time will be {\ttfamily (3,3)} and not {\ttfamily (2,2)} even though that\textquotesingle{}s the value received from the event.

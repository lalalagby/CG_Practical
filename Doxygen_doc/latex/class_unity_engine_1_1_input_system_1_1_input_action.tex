\doxysubsection{Detailed Description}
A named input signal that can flexibly decide which input data to tap. 

An input action is an abstraction over the source of input(s) it receives. They are most useful for representing input as "{}logical"{} concepts (e.\+g. "{}jump"{}) rather than as "{}physical"{} inputs (e.\+g. "{}space bar on keyboard pressed"{}).

In its most basic form, an action is simply an object along with a collection of bindings that trigger the action.

Bindings use control path expressions to reference controls. See \doxylink{struct_unity_engine_1_1_input_system_1_1_input_binding}{Input\+Binding} for more details. There may be arbitrary many bindings targeting a single action. The list of bindings targeting an action can be obtained through \doxylink{class_unity_engine_1_1_input_system_1_1_input_action_a44fdaf3ff9ffc22609a9df942372d9bb}{bindings}.

By itself an action does not do anything until it is enabled\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{action.Enable();}

\end{DoxyCode}
 

Once enabled, the action will actively monitor all controls on devices present in the system (see Input\+System.\+devices) that match any of the binding paths associated with the action. If you want to restrict the set of bindings used at runtime or restrict the set of devices which controls are chosen from, you can do so using \doxylink{class_unity_engine_1_1_input_system_1_1_input_action_a41caa3a44bc876781c5b1bcb04622cd2}{binding\+Mask} or, if the action is part of an \doxylink{class_unity_engine_1_1_input_system_1_1_input_action_map}{Input\+Action\+Map}, by setting the \doxylink{class_unity_engine_1_1_input_system_1_1_input_action_map_ab32ae5eb39f8c39dcebccb5cc1bb99e9}{Input\+Action\+Map.\+devices} property of the action map. The controls that an action uses can be queried using the \doxylink{class_unity_engine_1_1_input_system_1_1_input_action_a8e3f5d3329be13ca8474314137045af4}{controls} property.

When input is received on controls bound to an action, the action will trigger callbacks in response. These callbacks are \doxylink{class_unity_engine_1_1_input_system_1_1_input_action_a5fecf2acd129fa0922452ecf32527cdd}{started}, \doxylink{class_unity_engine_1_1_input_system_1_1_input_action_a11c046d8fb911d300e30e67827222c1f}{performed}, and \doxylink{class_unity_engine_1_1_input_system_1_1_input_action_a905374d6d7af6dc57a54acfa22018976}{canceled}. The callbacks are triggered as part of input system updates (see Input\+System.\+Update), i.\+e. they happen before the respective {\ttfamily Mono\+Behaviour.\+Update} or {\ttfamily Mono\+Behaviour.\+Fixed\+Update} methods get executed (depending on which \doxylink{class_unity_engine_1_1_input_system_1_1_input_settings_ab7d15182096158aaf407f0b7e9aed8ae}{Input\+Settings.\+update\+Mode} the system is set to).

In what order and how those callbacks get triggered depends on both the \doxylink{class_unity_engine_1_1_input_system_1_1_input_action_a4034bbebc91379913a9da5600a8a3491}{type} of the action as well as on the interactions (see \doxylink{interface_unity_engine_1_1_input_system_1_1_i_input_interaction}{IInput\+Interaction}) present on the bindings of the action. The default behavior is that when a control is actuated (that is, moving away from its resting position), \doxylink{class_unity_engine_1_1_input_system_1_1_input_action_a5fecf2acd129fa0922452ecf32527cdd}{started} is called and then \doxylink{class_unity_engine_1_1_input_system_1_1_input_action_a11c046d8fb911d300e30e67827222c1f}{performed}. Subsequently, whenever the a control further changes value to anything other than its default value, \doxylink{class_unity_engine_1_1_input_system_1_1_input_action_a11c046d8fb911d300e30e67827222c1f}{performed} will be called again. Finally, when the control moves back to its default value (i.\+e. resting position), \doxylink{class_unity_engine_1_1_input_system_1_1_input_action_a905374d6d7af6dc57a54acfa22018976}{canceled} is called.

To hook into the callbacks, there are several options available to you. Enable the action such that it actively listens for input and runs callbacks in response. 

If the action is already enabled, this method does nothing.

By default, actions start out disabled, i.\+e. with \doxylink{class_unity_engine_1_1_input_system_1_1_input_action_a10b7ab479ef1419726f2d05927fd1b70}{enabled} being false. When enabled, two things happen.

First, if it hasn\textquotesingle{}t already happened, an action will resolve all of its bindings to \doxylink{class_unity_engine_1_1_input_system_1_1_input_control}{Input\+Control}s. This also happens if, since the action was last enabled, the setup of devices in the system has changed such that it may impact the action.

Second, for all the \doxylink{class_unity_engine_1_1_input_system_1_1_input_action_a8e3f5d3329be13ca8474314137045af4}{controls} bound to an action, change monitors (see IInput\+State\+Change\+Monitor) will be added to the system. If any of the controls changes state in the future, the action will get notified and respond.

\doxylink{namespace_unity_engine_1_1_input_system_a0d0d044aae0aef15123e2f31f11ea94ba689202409e48743b914713f96d93947c}{Input\+Action\+Type.\+Value} type actions will also perform an initial state check in the input system update following the call to Enable. This means that if any of the bound controls are already actuated and produce a non-\/{\ttfamily default} value, the action will immediately trigger in response.

Note that this method only enables a single action. This is also allowed for action that are part of an \doxylink{class_unity_engine_1_1_input_system_1_1_input_action_map}{Input\+Action\+Map}. To enable all actions in a map, call \doxylink{class_unity_engine_1_1_input_system_1_1_input_action_map_a1070fcb5a4631ca6c0212fb28a19644f}{Input\+Action\+Map.\+Enable}.

The \doxylink{class_unity_engine_1_1_input_system_1_1_input_action_map}{Input\+Action\+Map} associated with an action (if any), will immediately toggle to being enabled (see \doxylink{class_unity_engine_1_1_input_system_1_1_input_action_map_a178f645fe8f329485cff4cdb3e69086c}{Input\+Action\+Map.\+enabled}) as soon as the first action in the map is enabled and for as long as any action in the map is still enabled.

The first time an action is enabled, it will allocate a block of state internally that it will hold on to until disposed of. For free-\/standing actions, that state is private to just the action. For actions that are part of \doxylink{class_unity_engine_1_1_input_system_1_1_input_action_map}{Input\+Action\+Map}s, the state is shared by all actions in the map and, if the map itself is part of an \doxylink{class_unity_engine_1_1_input_system_1_1_input_action_asset}{Input\+Action\+Asset}, also by all the maps that are part of the asset.

To dispose of the state, call \doxylink{class_unity_engine_1_1_input_system_1_1_input_action_a5c45bb13be7dafd32a3cfd23ba85a80e}{Dispose}.

If, say, you have a \doxylink{class_unity_engine_1_1_input_system_1_1_interactions_1_1_hold_interaction}{Interactions.\+Hold\+Interaction} on a binding, you might want to show a progress indicator in the \doxylink{namespace_unity_engine_1_1_u_i}{UI} and need to know how far into the hold the action current is. Once the hold has been started, this method will return how far into the hold the action currently is.

Note that if an interaction performs and stays performed (see Input\+Interaction\+Context.\+Performed\+And\+Stay\+Performed), the completion percentage will remain at 1 until the interaction is canceled.

Also note that completion is based on the progression of time and not dependent on input updates. This means that if, for example, the timeout for a \doxylink{class_unity_engine_1_1_input_system_1_1_interactions_1_1_hold_interaction}{Interactions.\+Hold\+Interaction} has expired according the current time but the expiration has not yet been processed by an input update (thus causing the hold to perform), the returned completion percentage will still be 1. In other words, there isn\textquotesingle{}t always a correlation between the current completion percentage and \doxylink{class_unity_engine_1_1_input_system_1_1_input_action_a86bd545c9dfe1fb0200e296c44389e57}{phase}.

The meaning of the timeout is dependent on the interaction in play. For a \doxylink{class_unity_engine_1_1_input_system_1_1_interactions_1_1_hold_interaction}{Interactions.\+Hold\+Interaction}, the timeout represents "{}completion"{} (that is, the time until a "{}hold"{} is considered to be performed), whereas for a \doxylink{class_unity_engine_1_1_input_system_1_1_interactions_1_1_tap_interaction}{Interactions.\+Tap\+Interaction} it represents "{}time to failure"{} (that is, the remaining time window that the interaction can be completed within).

Note that an interaction might run multiple timeouts in succession. One such example is \doxylink{class_unity_engine_1_1_input_system_1_1_interactions_1_1_multi_tap_interaction}{Interactions.\+Multi\+Tap\+Interaction}. In this case, progression towards a single timeout does not necessarily mean progression towards completion of the whole interaction. An interaction can call \doxylink{struct_unity_engine_1_1_input_system_1_1_input_interaction_context_a5284a68e9a6129992bc31890d5d173cc}{Input\+Interaction\+Context.\+Set\+Total\+Timeout\+Completion\+Time} to inform the Input \doxylink{namespace_system}{System} of the total length of timeouts to run. If this is done, the result of the {\ttfamily Get\+Timeout\+Completion\+Percentage} method will return a value reflecting the progression with respect to total time.

Only after the hold time has expired will {\ttfamily Was\+Performed\+This\+Frame} be true and only in the frame where the hold performed.

This is different from checking \doxylink{class_unity_engine_1_1_input_system_1_1_input_action_a86bd545c9dfe1fb0200e296c44389e57}{phase} directly as the action might have already progressed to a different phase after performing. \mbox{\hyperlink{struct_unity_engine_1_1_input_system_1_1_input_binding}{Input\+Binding}}? Unity\+Engine.\+Input\+System.\+Input\+Action.\+binding\+Mask\hspace{0.3cm}{\ttfamily [get]}, {\ttfamily [set]}}



An optional mask that determines which bindings of the action to enable and which to ignore. 

Optional mask that determines which bindings on the action to enable.

Binding masks can be applied at three different levels\+: for an entire asset through \doxylink{class_unity_engine_1_1_input_system_1_1_input_action_asset_a7cf65e9b19155bd044d9d06250e61225}{Input\+Action\+Asset.\+binding\+Mask}, for a specific map through \doxylink{class_unity_engine_1_1_input_system_1_1_input_action_map_af68ee392460d425649d96935d8f7345d}{Input\+Action\+Map.\+binding\+Mask}, and for single actions through this property. By default, none of the masks will be set (i.\+e. they will be {\ttfamily null}).

When an action is enabled, all the binding masks that apply to it are taken into account. Specifically, this means that any given binding on the action will be enabled only if it matches the mask applied to the asset, the mask applied to the map that contains the action, and the mask applied to the action itself. All the masks are individually optional.

Masks are matched against bindings using \doxylink{struct_unity_engine_1_1_input_system_1_1_input_binding_aa69fd73f446257c84723c95ad8fbcbe8}{Input\+Binding.\+Matches}.

Note that if you modify the masks applicable to an action while it is enabled, the action\textquotesingle{}s \doxylink{class_unity_engine_1_1_input_system_1_1_input_action_a8e3f5d3329be13ca8474314137045af4}{controls} will get updated immediately to respect the mask. To avoid repeated binding resolution, it is most efficient to apply binding masks before enabling actions.

Binding masks are non-\/destructive. All the bindings on the action are left in place. Setting a mask will not affect the value of the \doxylink{class_unity_engine_1_1_input_system_1_1_input_action_a44fdaf3ff9ffc22609a9df942372d9bb}{bindings} property.

This property can be queried whether the action is enabled or not and will return the set of controls that match the action\textquotesingle{}s bindings according to the current setup of binding masks (\doxylink{class_unity_engine_1_1_input_system_1_1_input_action_a41caa3a44bc876781c5b1bcb04622cd2}{binding\+Mask}) and device restrictions (\doxylink{class_unity_engine_1_1_input_system_1_1_input_action_map_ab32ae5eb39f8c39dcebccb5cc1bb99e9}{Input\+Action\+Map.\+devices}).

Note that internally, controls are not stored on a per-\/action basis. This means that on the first read of this property, the list of controls for just the action may have to be extracted which in turn may allocate GC memory. After the first read, no further GC allocations should occur except if the set of controls is changed (e.\+g. by changing the binding mask or by adding/removing devices to/from the system).

If the property is queried when the action has not been enabled yet, the system will first resolve controls on the action (and for all actions in the map and/or the asset). See \href{../manual/ActionBindings.html\#binding-resolution}{\texttt{ Binding Resolution}} in the manual for details.

Usually, when an action is \doxylink{class_unity_engine_1_1_input_system_1_1_input_action_a10b7ab479ef1419726f2d05927fd1b70}{enabled} (e.\+g. via \doxylink{class_unity_engine_1_1_input_system_1_1_input_action_abc6ff7728cbf947406b3c426cf82437f}{Enable}), it will start listening for input and then trigger once the first input arrives. However, \doxylink{class_unity_engine_1_1_input_system_1_1_input_action_a8e3f5d3329be13ca8474314137045af4}{controls} bound to an action may already be actuated when an action is enabled. For example, if a "{}jump"{} action is bound to \doxylink{class_unity_engine_1_1_input_system_1_1_keyboard_a29e179bc8d284cf71be6847ca6ae77f3}{Keyboard.\+space\+Key}, the space bar may already be pressed when the jump action is enabled.

\doxylink{namespace_unity_engine_1_1_input_system_a0d0d044aae0aef15123e2f31f11ea94ba689202409e48743b914713f96d93947c}{Input\+Action\+Type.\+Value} actions handle this differently by immediately performing an "{}initial state check"{} in the next input update (see Input\+System.\+Update) after being enabled. If any of the bound controls is already actuated, the action will trigger right away -- even with no change in state on the controls.

This same behavior can be enabled explicitly for \doxylink{namespace_unity_engine_1_1_input_system_a0d0d044aae0aef15123e2f31f11ea94ba87b7760f14fbff78d8819291f36ab9a0}{Input\+Action\+Type.\+Button} and \doxylink{namespace_unity_engine_1_1_input_system_a0d0d044aae0aef15123e2f31f11ea94bae1ff6a7360e99cb46db6bb5d3ff8e2eb}{Input\+Action\+Type.\+Pass\+Through} actions using this property.

An \href{../api/UnityEngine.InputSystem.InputBinding.html}{\texttt{ {\ttfamily Input\+Binding}}} represents a connection between an \doxysectlink{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_actions}{Action}{0} and one or more \doxysectlink{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_controls}{Controls}{0} identified by a \href{Controls.md\#control-paths}{\texttt{ Control path}}. An Action can have an arbitrary number of Bindings pointed at it. Multiple Bindings can reference the same Control.

Sometimes, you might want to have several Controls act in unison to mimic a different type of Control. The most common example of this is using the W, A, S, and D keys on the keyboard to form a 2D vector Control equivalent to mouse deltas or gamepad sticks. Another example is to use two keys to form a 1D axis equivalent to a mouse scroll axis.

This is difficult to implement with normal Bindings. You can bind a \href{../api/UnityEngine.InputSystem.Controls.ButtonControl.html}{\texttt{ {\ttfamily Button\+Control}}} to an action expecting a {\ttfamily Vector2}, but doing so results in an exception at runtime when the Input \doxylink{namespace_system}{System} tries to read a {\ttfamily Vector2} from a Control that can deliver only a {\ttfamily float}.

Composite Bindings (that is, Bindings that are made up of other Bindings) solve this problem. Composites themselves don\textquotesingle{}t bind directly to Controls; instead, they source values from other Bindings that do, and then synthesize input on the fly from those values.

To see how to create Composites in the editor UI, see documentation on \href{ActionAssets.md\#editing-composite-bindings}{\texttt{ editing Composite Bindings}}.

To create composites in code, you can use the \href{../api/UnityEngine.InputSystem.InputActionSetupExtensions.html\#UnityEngine_InputSystem_InputActionSetupExtensions_AddCompositeBinding_UnityEngine_InputSystem_InputAction_System_String_System_String_System_String_}{\texttt{ {\ttfamily Add\+Composite\+Binding}}} syntax.

Each Composite consists of one Binding that has \href{../api/UnityEngine.InputSystem.InputBinding.html\#UnityEngine_InputSystem_InputBinding_isComposite}{\texttt{ {\ttfamily Input\+Binding.\+is\+Composite}}} set to true, followed by one or more Bindings that have \href{../api/UnityEngine.InputSystem.InputBinding.html\#UnityEngine_InputSystem_InputBinding_isPartOfComposite}{\texttt{ {\ttfamily Input\+Binding.\+is\+Part\+Of\+Composiste}}} set to true. In other words, several consecutive entries in \href{../api/UnityEngine.InputSystem.InputActionMap.html\#UnityEngine_InputSystem_InputActionMap_bindings}{\texttt{ {\ttfamily Input\+Action\+Map.\+bindings}}} or \href{../api/UnityEngine.InputSystem.InputAction.html\#UnityEngine_InputSystem_InputAction_bindings}{\texttt{ {\ttfamily Input\+Action.\+bindings}}} together form a Composite.

Note that each composite part can be bound arbitrary many times.

To alter the current value of a parameter, you can use what is referred to as a "{}parameter override"{}. You can apply these at the level of an individual \href{../api/UnityEngine.InputSystem.InputAction.html}{\texttt{ {\ttfamily Input\+Action}}}, or at the level of an entire \href{../api/UnityEngine.InputSystem.InputActionMap.html}{\texttt{ {\ttfamily Input\+Action\+Map}}}, or even at the level of an entire \href{../api/UnityEngine.InputSystem.InputActionAsset.html}{\texttt{ {\ttfamily Input\+Action\+Asset}}}. Such overrides are stored internally and applied automatically even on bindings added later.

To add an override, use the \href{../api/UnityEngine.InputSystem.InputActionRebindingExtensions.html\#UnityEngine_InputSystem_InputActionRebindingExtensions_ApplyParameterOverride_UnityEngine_InputSystem_InputAction_System_String_UnityEngine_InputSystem_Utilities_PrimitiveValue_UnityEngine_InputSystem_InputBinding_}{\texttt{ {\ttfamily Apply\+Parameter\+Override}}} API or any of its overloads.

A Binding can belong to any number of Binding groups. \doxylink{namespace_unity}{Unity} stores these on the \href{../api/UnityEngine.InputSystem.InputBinding.html}{\texttt{ {\ttfamily Input\+Binding}}} class as a semicolon-\/separated string in the \href{../api/UnityEngine.InputSystem.InputBinding.html\#UnityEngine_InputSystem_InputBinding_groups}{\texttt{ {\ttfamily Input\+Binding.\+groups}}} property, and you can use them for any arbitrary grouping of bindings. To enable different sets of binding groups for an \href{../api/UnityEngine.InputSystem.InputActionMap.html}{\texttt{ {\ttfamily Input\+Action\+Map}}} or \href{../api/UnityEngine.InputSystem.InputActionAsset.html}{\texttt{ {\ttfamily Input\+Action\+Asset}}}, you can use the \href{../api/UnityEngine.InputSystem.InputActionMap.html\#UnityEngine_InputSystem_InputActionMap_bindingMask}{\texttt{ {\ttfamily Input\+Action\+Map.\+binding\+Mask}}}/\href{../api/UnityEngine.InputSystem.InputActionAsset.html\#UnityEngine_InputSystem_InputActionAsset_bindingMask}{\texttt{ {\ttfamily Input\+Action\+Asset.\+binding\+Mask}}} property. The Input \doxylink{namespace_system}{System} uses this to implement the concept of grouping Bindings into different \href{../api/UnityEngine.InputSystem.InputControlScheme.html}{\texttt{ {\ttfamily Input\+Control\+Schemes}}}.

Control Schemes use Binding groups to map Bindings in an \href{../api/UnityEngine.InputSystem.InputActionMap.html}{\texttt{ {\ttfamily Input\+Action\+Map}}} or \href{../api/UnityEngine.InputSystem.InputActionAsset.html}{\texttt{ {\ttfamily Input\+Action\+Asset}}} to different types of Devices. When the Input \doxylink{namespace_system}{System} accesses the \doxysectlink{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_controls}{Controls}{0} bound to an Action for the first time, the Action resolves its Bindings to match them to existing Controls on existing Devices. In this process, the Action calls \href{../api/UnityEngine.InputSystem.InputSystem.html\#UnityEngine_InputSystem_InputSystem_FindControls__1_System_String_UnityEngine_InputSystem_InputControlList___0___}{\texttt{ {\ttfamily Input\+System.\+Find\+Controls\texorpdfstring{$<$}{<}\texorpdfstring{$>$}{>}()}}} (filtering for devices assigned to the Input\+Action\+Map, if there are any) for the Binding path of each of the Action\textquotesingle{}s bindings. This creates a list of resolved Controls that are now bound to the Action.

Note that a single \href{Controls.md\#control-paths}{\texttt{ Binding path}} can match multiple Controls\+:


\begin{DoxyItemize}
\item A specific Device path such as {\ttfamily \texorpdfstring{$<$}{<}Dual\+Shock\+Gamepad\texorpdfstring{$>$}{>}/button\+East} matches the "{}\+Circle"{} button on a \href{Gamepad.md\#playstation-controllers}{\texttt{ Play\+Station controller}}. If you have multiple Play\+Station controllers connected, it resolves to the "{}\+Circle"{} button on each of these controllers.
\item An abstract Device path such as {\ttfamily \texorpdfstring{$<$}{<}Gamepad\texorpdfstring{$>$}{>}/button\+East} matches the right action button on any connected gamepad. If you have a Play\+Station controller and an \href{Gamepad.md\#xbox-controllers}{\texttt{ Xbox controller}} connected, it resolves to the "{}\+Circle"{} button on the Play\+Station controller, and to the "{}\+B"{} button on the Xbox controller.
\item A Binding path can also contain wildcards, such as {\ttfamily \texorpdfstring{$<$}{<}Gamepad\texorpdfstring{$>$}{>}/button\texorpdfstring{$\ast$}{*}}. This matches any Control on any gamepad with a name starting with "{}button"{}, which matches all the four action buttons on any connected gamepad. A different example\+: {\ttfamily \texorpdfstring{$\ast$}{*}/\{Submit\}} matches any Control tagged with the "{}\+Submit"{} \href{Controls.md\#control-usages}{\texttt{ usage}} on any Device.
\end{DoxyItemize}

If there are multiple Bindings on the same Action that all reference the same Control(s), the Control will effectively feed into the Action multiple times. This is to allow, for example, a single Control to produce different input on the same Action by virtue of being bound in a different fashion (composites, processors, interactions, etc). However, regardless of how many times a Control is bound on any given action, it will only be mentioned once in the Action\textquotesingle{}s \href{../api/UnityEngine.InputSystem.InputAction.html\#UnityEngine_InputSystem_InputAction_controls}{\texttt{ array of {\ttfamily controls}}}.

To query the Controls that an Action resolves to, you can use \href{../api/UnityEngine.InputSystem.InputAction.html\#UnityEngine_InputSystem_InputAction_controls}{\texttt{ {\ttfamily Input\+Action.\+controls}}}. You can also run this query if the Action is disabled.

In certain situations, the \href{../api/UnityEngine.InputSystem.InputAction.html\#UnityEngine_InputSystem_InputAction_controls}{\texttt{ Controls}} bound to an Action have to be updated more than once. For example, if a new \doxysectlink{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_devices}{Device}{0} becomes usable with an Action, the Action may now pick up input from additional controls. Also, if Bindings are added, removed, or modified, Control lists will need to be updated.

This updating of Controls usually happens transparently in the background. However, when an Action is \href{../api/UnityEngine.InputSystem.InputAction.html\#UnityEngine_InputSystem_InputAction_enabled}{\texttt{ enabled}} and especially when it is \href{../api/UnityEngine.InputSystem.InputAction.html\#UnityEngine_InputSystem_InputAction_IsInProgress_}{\texttt{ in progress}}, there may be a noticeable effect on the Action.

Adding or removing a device -- either \href{../api/UnityEngine.InputSystem.InputSystem.html\#UnityEngine_InputSystem_InputSystem_devices}{\texttt{ globally}} or to/from the \href{../api/UnityEngine.InputSystem.InputActionAsset.html\#UnityEngine_InputSystem_InputActionAsset_devices}{\texttt{ device list}} of an Action -- will remain transparent {\bfseries{except}} if an Action is in progress and it is the device of its \href{../api/UnityEngine.InputSystem.InputAction.html\#UnityEngine_InputSystem_InputAction_activeControl}{\texttt{ active Control}} that is being removed. In this case, the Action will automatically be \href{../api/UnityEngine.InputSystem.InputAction.html\#UnityEngine_InputSystem_InputAction_canceled}{\texttt{ cancelled}}.

Modifying the \href{../api/UnityEngine.InputSystem.InputActionAsset.html\#UnityEngine_InputSystem_InputActionAsset_bindingMask}{\texttt{ binding mask}} or modifying any of the Bindings (such as through rebinding or by adding or removing bindings) will, however, lead to all enabled Actions being temporarily disabled and then re-\/enabled and resumed.\hypertarget{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_action_bindings_autotoc_md1260}{}\doxysubsubsection{\texorpdfstring{Choosing which Devices to use}{Choosing which Devices to use}}\label{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_action_bindings_autotoc_md1260}

There are two situations where a given input may lead to ambiguity\+:


\begin{DoxyEnumerate}
\item Several Controls are bound to the same Action and more than one is feeding input into the Action at the same time. Example\+: an Action that is bound to both the left and right trigger on a Gamepad and both triggers are pressed.
\item The input is part of a sequence of inputs and there are several possible such sequences. Example\+: one Action is bound to the {\ttfamily B} key and another Action is bound to {\ttfamily Shift-\/B}.
\end{DoxyEnumerate}\hypertarget{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_action_bindings_autotoc_md1262}{}\doxysubsubsection{\texorpdfstring{Multiple, concurrently used Controls}{Multiple, concurrently used Controls}}\label{md__hey_tea_9_2_library_2_package_cache_2com_8unity_8inputsystem_0d1_86_83_2_documentation_0i_2_action_bindings_autotoc_md1262}
\texorpdfstring{$>$}{>}{\bfseries{Note\+:}} This section does not apply to \href{Actions.md\#pass-through}{\texttt{ {\ttfamily Pass\+Through}}} Actions as they are by design meant to allow multiple concurrent inputs.

For a \href{Actions.md\#button}{\texttt{ {\ttfamily Button}}} or \href{Actions.md\#value}{\texttt{ {\ttfamily Value}}} Action, there can only be one Control at any time that is "{}driving"{} the Action. This Control is considered the \href{../api/UnityEngine.InputSystem.InputAction.html\#UnityEngine_InputSystem_InputAction_activeControl}{\texttt{ {\ttfamily active\+Control}}}.

When an Action is bound to multiple Controls, the \href{../api/UnityEngine.InputSystem.InputAction.html\#UnityEngine_InputSystem_InputAction_activeControl}{\texttt{ {\ttfamily active\+Control}}} at any point is the one with the greatest level of \href{Controls.md\#control-actuation}{\texttt{ "{}actuation"{}}}, that is, the largest value returned from \href{../api/UnityEngine.InputSystem.InputControl.html\#UnityEngine_InputSystem_InputControl_EvaluateMagnitude_}{\texttt{ {\ttfamily Evaluate\+Magnitude}}}. If a Control exceeds the actuation level of the current \href{../api/UnityEngine.InputSystem.InputAction.html\#UnityEngine_InputSystem_InputAction_activeControl}{\texttt{ {\ttfamily active\+Control}}}, it will itself become the active Control.

Inputs that are used in combinations with other inputs may also lead to ambiguities. If, for example, the {\ttfamily b} key on the Keyboard is bound both on its own as well as in combination with the {\ttfamily shift} key, then if you first press {\ttfamily shift} and then {\ttfamily b}, the latter key press would be a valid input for either of the Actions.

The way this is handled is that Bindings will be processed in the order of decreasing "{}complexity"{}. This metric is derived automatically from the Binding\+:


\begin{DoxyItemize}
\item A binding that is {\itshape not} part of a composite is assigned a complexity of 1.
\item A binding that {\itshape is} part of a composite is assigned a complexity equal to the number of part bindings in the composite.
\end{DoxyItemize}

In our example, this means that a \`{}\+One\+Modifier\`{} composite Binding to {\ttfamily Shift+B} has a higher "{}complexity"{} than a Binding to {\ttfamily B} and thus is processed first.

Additionally, the first Binding that results in the Action changing \href{Actions.md\#action-callbacks}{\texttt{ phase}} will "{}consume"{} the input. This consuming will result in other Bindings to the same input not being processed. So in our example, when {\ttfamily Shift+B} "{}consumes"{} the {\ttfamily B} input, the Binding to {\ttfamily B} will be skipped.

After an Action is \href{../api/UnityEngine.InputSystem.InputAction.html\#UnityEngine_InputSystem_InputAction_enabled}{\texttt{ enabled}}, it will start reacting to input as it comes in. However, at the time the Action is enabled, one or more of the Controls that are \href{../api/UnityEngine.InputSystem.InputAction.html\#UnityEngine_InputSystem_InputAction_controls}{\texttt{ bound}} to an action may already have a non-\/default state at that point.

Using what is referred to as an "{}initial state check"{}, an Action can be made to respond to such a non-\/default state as if the state change happened {\itshape after} the Action was enabled. The way this works is that in the first input \href{../api/UnityEngine.InputSystem.InputSystem.html\#UnityEngine_InputSystem_InputSystem_Update_}{\texttt{ update}} after the Action was enabled, all its bound controls are checked in turn. If any of them has a non-\/default state, the Action responds right away.

This check is implicitly enabled for \href{Actions.md\#value}{\texttt{ Value}} actions. If, for example, you have a {\ttfamily Move} Action bound to the left stick on the gamepad and the stick is already pushed in a direction when {\ttfamily Move} is enabled, the character will immediately start walking.

By default, \href{Actions.md\#button}{\texttt{ Button}} and \href{Actions.md\#pass-through}{\texttt{ Pass-\/\+Through}} type Actions, do not perform this check. A button that is pressed when its respective Action is enabled first needs to be released and then pressed again for it to trigger the Action.
